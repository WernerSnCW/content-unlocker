artifacts/api-server/build.mjs
  Custom esbuild bundler. Resolves __dirname via fileURLToPath(import.meta.url),
  recreates dist/, runs esbuild on src/index.ts → dist/index.mjs (ESM,
  platform=node, bundle=true, outExtension .js→.mjs), externalises every
  native module known to be unbundleable (sharp, better-sqlite3, sqlite3,
  canvas, bcrypt, argon2, fsevents, pg-native, oracledb, etc., plus the
  *.node loader extension and a long allowlist).

artifacts/api-server/src/index.ts
  Boot script. Reads PORT (throws if missing or non-numeric). Runs
  `seedDatabase()` from lib/dataManager, then `loadConstants()` from
  lib/complianceConstantsService — if zero compliance constants come
  back the process logs FATAL and exits(1). Calls `startUntaggedSweep()`
  (a 5-minute interval defined in routes/aircall) and finally calls
  `app.listen(port)`. On seed failure it logs and continues to listen
  anyway.

artifacts/api-server/src/app.ts
  Express 5 app factory. trust proxy = 1. Mounts pino-http with custom
  req/res serialisers. CORS: if APP_URL env is set → restrict origin to
  it with credentials:true; otherwise permissive cors() (early dev).
  Mounts cookieParser, express.json, express.urlencoded({extended:true}).
  Wraps `buildSessionMiddleware()` in try/catch — if SESSION_SECRET is
  missing the session middleware is skipped and a warning is logged
  (only /api/auth/* is documented to fail in that case). Finally
  mounts the api router under `/api`.

artifacts/api-server/src/routes/index.ts
  Single barrel router that imports and mounts (in this order):
  authRouter, adminRouter, healthRouter, leadsRouter, documentsRouter,
  recommendationRouter, contentRouter, generationRouter, dashboardRouter,
  gdocsRouter, callFrameworkRouter, analyticsRouter, videosRouter,
  acuRouter, campaignsRouter, templatesRouter, promptsRouter,
  complianceConstantsRouter, importRouter, tasksRouter,
  sheetSyncRouter (mounted at `/sheet-sync`), beliefsRouter,
  leadBeliefsRouter, leadIntelligenceRouter, workQueueRouter,
  documentHealthRouter, settingsRouter, contactsRouter,
  campaignsDispatchRouter, aircallRouter, engineRouter,
  outcomeReviewsRouter, eventsRouter.

artifacts/api-server/src/middlewares/requireAuth.ts
  Reads req.session.userId, loads usersTable + agentsTable, attaches
  {user, agent} to req.auth, or responds 401/403 (not_authenticated,
  user_not_found, no_agent_linked, agent_inactive). On user_not_found
  it destroys the session.

artifacts/api-server/src/middlewares/requireAdmin.ts
  Composes requireAuth and then verifies req.auth.user.role === "admin",
  otherwise 403 admin_only.

artifacts/api-server/src/lib/auth/oidcClient.ts
  Google OIDC client. Caches `discovery()` for issuer
  https://accounts.google.com using GOOGLE_CLIENT_ID, GOOGLE_CLIENT_SECRET,
  APP_URL (throws if any missing). Builds redirect_uri =
  `${APP_URL}/api/auth/callback`. Optional GOOGLE_WORKSPACE_DOMAIN passed as
  hd hint. Exports getOidcContext, buildAuthorizationUrl (PKCE S256 +
  state + nonce, scope `openid email profile`), exchangeCode (validates
  ID token via openid-client.authorizationCodeGrant).

artifacts/api-server/src/lib/auth/session.ts
  buildSessionMiddleware: express-session backed by connect-pg-simple over
  the pg.Pool exported from @workspace/db. Cookie name `cu.sid`, rolling
  sliding 7-day maxAge, httpOnly, sameSite=lax, secure when
  NODE_ENV==='production'. createTableIfMissing is intentionally NOT used
  (the session table is declared in lib/db schema). Validates SESSION_SECRET
  is present and ≥32 chars.

artifacts/api-server/src/lib/queueEvents.ts
  In-process SSE broadcaster for call-queue updates. Webhook handlers and
  the untagged-sweep call notifyQueueChanged() to push a refresh signal to
  every connected Call Command page. Single-process only.

artifacts/api-server/src/lib/logger.ts

  src/data/. Maintains a hardcoded PROHIBITED_VALUES guard
  ("22p","7.8x","78x") used by validateOverride.

  3577-line OpenAPI 3.1.0 spec. info.title="Api", version 0.1.0.
  servers: /api. Tag list (header confirmed) includes: health, leads,
  recommendation, documents, compliance, changelog, content, gaps,
  feature-update, generation, dashboard, … (full tag list and
  operations not enumerated — only the file header).

lib/api-spec/orval.config.ts
  Two Orval generators driven from openapi.yaml:
    "api-client-react" → lib/api-client-react/src/generated, mode "split",
      client "react-query", baseUrl "/api", custom-fetch mutator at
      lib/api-client-react/src/custom-fetch.ts.
    "zod"              → lib/api-zod/src/generated, mode "split",
      client "zod", schemas under generated/types, useDates+useBigInt,
      coerce config covers boolean/number/string for query+param,
      bigint+date for body+response.
  An InputTransformerFn forces info.title="Api" so the generated
  module name is api.ts.

lib/api-spec/package.json
  @workspace/api-spec, scripts: codegen (`orval --config ./orval.config.ts`).

lib/api-zod/
  package.json + tsconfig.json + dist/ + src/index.ts (wrapper) +
  src/generated/ (Orval output — files not read individually).

lib/api-client-react/
  package.json + tsconfig.json + dist/ + src/index.ts +
  src/custom-fetch.ts (Orval mutator) + src/generated/ (Orval output).

lib/db/package.json
  @workspace/db. exports: . → src/index.ts, ./schema → src/schema/index.ts.
  scripts: push (`drizzle-kit push --config ./drizzle.config.ts`),
  push-force.
  deps: drizzle-orm (catalog), drizzle-zod ^0.8.3, pg ^8.20.0,
  zod (catalog). devDep: drizzle-kit ^0.31.9.

lib/db/drizzle.config.ts
  Reads DATABASE_URL (throws if missing). dialect "postgresql", schema
  path = ./src/schema/index.ts.

lib/db/src/index.ts
  Builds a `pg.Pool({connectionString: DATABASE_URL})` (throws if
  DATABASE_URL is unset), exports it as `pool`, builds and exports the
  Drizzle ORM instance as `db = drizzle(pool, {schema})`. Re-exports
  everything from ./schema.

lib/db/src/schema/index.ts
  Barrel that re-exports all 41 schema files (full list in Section 2).

lib/db/src/schema/  — 44 files total, comprising 41 table modules + the
barrel + the runtime session table + the document health pair. Each
module follows the same pattern (drizzle-orm pgTable, drizzle-zod
insertSchema, exported InsertX/X types). Per-table column inventory in
Section 2.

SECTION 3 — DATA FLOW (input → storage → display)
===========================================================================

Below describes the flows that were directly observable from the code
read in this scan. Where a flow runs through a route handler whose body
was not opened in detail, the description is limited to what the route
mounting, schema definitions, and file headers establish.

A. AUTHENTICATION (read end-to-end)
   User → /login (frontend page) → user clicks "Sign in with Google" →
   GET /api/auth/google → buildAuthorizationUrl() in
   lib/auth/oidcClient.ts generates state/PKCE/nonce, stores them on
   req.session, redirects browser to Google →
   Google redirects back to GET /api/auth/callback with code →
   exchangeCode() validates ID token (state, nonce, PKCE) →
   handler upserts users row (matched by google_sub), looks up agents
   row by user_id (binding established at first login by email match),
   sets req.session.userId and persists.
   On every authed request, requireAuth loads usersTable + agentsTable
   into req.auth. requireAdmin further checks user.role === 'admin'.
   Frontend: ProtectedRoute calls useCurrentUser → /api/auth/me which
   returns the {user, agent} payload; on 401 the route redirects to
   /login?returnTo=<path>. Logout: POST /api/auth/logout, then
   apiClient.logout() forces window.location to /login.
   Storage: users (Postgres), agents (Postgres), session (Postgres via
   connect-pg-simple in the `session` table).

B. CALL TRANSCRIPT INTAKE (structural — handler bodies not all read)
   Two ingress paths mention this flow in code that was opened:
     1. Aircall webhook: POST /api/aircall/webhook (handlers in
        routes/aircall/index.ts; bodies not read).
     2. Sheet sync: POST /api/sheet-sync/sync (routes/sheet-sync;
        bodies not read).
   In both cases lead_conversations rows are created (schema confirms
   source ∈ aircall|fireflies|manual) and the engine pipeline is
   invoked.
   Engine entry (verified via routes/engine/index.ts imports):
     loadInvestor(contactId) → processTranscript / processTranscriptWithLLM
     / processTranscriptDetailed (engine/v2) → saveEngineRun.
   Persistence side-effects per saveEngineRun (per file headers and
   schema):
     engine_runs row written (call_type, engine_version, full output
       jsonb, llm_* audit columns, status one of ok|failed|keyword).
     engine_signals upserted per (contact, code).
     engine_signal_transitions appended for state changes.
     engine_investor_state upserted per contact.
   When the call.tagged path detects an outcome configured with
   creates_outcome_review=true (per the comment in
   outcome_reviews schema), an outcome_reviews row is created in the
   same transaction; outcome_action_decisions rows are then written
   per operator action via the outcome-reviews router.
   When content is actually dispatched (separate event from the engine
   recommendation), engine_artifacts_sent rows are written.

C. CONTACT INGESTION (structural)
   POST /api/contacts/uploads creates an upload_sessions row →
   /uploads/suggest + /uploads/:id/decisions populate staged_contacts
   with dedup_status (new|exact_duplicate|possible_match|invalid) →
   POST /uploads/:id/commit promotes accepted rows into the contacts
   table (contact_id `contact_<uuid8>`). The contactIngestionService.ts
   header confirms this is its scope.

D. CALL LIST DISPATCH (structural)
   Admins manage call_list_configs (filters + daily quota). Membership
   is tracked in call_list_memberships with a partial UNIQUE INDEX on
   contact_id WHERE removed_at IS NULL — at most one active membership
   per contact at any time. Tiered fillQueue is invoked via
   POST /api/call-lists/:id/fill-queue; the closing_only flag scopes
   to tier-0 conversion calls only and is meaningful only for closer
   role agents (per the schema comment).
   Power Dialer push: POST /api/call-lists/:id/send-to-power-dialer
   plus admin-side queue inspection
   GET /api/admin/agents/:id/power-dialer-queue and clear endpoint.

E. NEXT-BEST-ACTION DECISION (structural)
   determineNextAction (engine/v2/functions/) is the legacy hardcoded
   cascade. The Phase 7.1a outcome-rules engine
   (engine/v2/outcomeRules/{loader, evaluator}) reads
   engine_outcome_rules rows in priority order, evaluates the
   AND-clause set against an EvaluationContext, and returns the matching
   rule's actions list. Gating env: ENGINE_OUTCOME_RULES.
   Admin endpoints (admin/index.ts):
     GET    /admin/engine-outcome-rules
     POST   /admin/engine-outcome-rules + /seed
     PUT    /admin/engine-outcome-rules/:id
     DELETE /admin/engine-outcome-rules/:id
     POST   /admin/engine-outcome-rules/trace
     POST   /admin/engine-outcome-rules/preview
   Frontend pages /admin/outcome-rules + /admin/engine-config display
   and edit these rules.

F. DOCUMENT REGISTRY → PDF / GOOGLE DOC (structural)
   Documents are stored in the documents table. PATCH /documents/:id
   writes content_updated_at. POST /documents/:id/generate-pdf renders
   via lib/pdfService.ts (Puppeteer-core + Inter font + Design Bible
   HTML template) and persists pdf_exported_at + pdf_file_path.
   GET /documents/:id/download-pdf serves the cached PDF.
   POST /gdocs/export/:id pushes content to Google Drive/Docs; the
   formatter pipeline in lib/formatService.ts (Claude with the
   Design Bible v0.2 system prompt) is what produces the HTML body
   that's uploaded.

G. CONTENT GENERATION (structural)
   Generation pipeline: POST /api/generation/generate (and /from-template,
   /:id/regenerate, /:id/qc-rerun, /:id/promote) — orchestrated by
   lib/generationEngine.ts. Per-section compliance enforcement
   (max_words, max_sentences, max_chars, locked-content injection,
   prohibited-ACU exclusion) lives in checkTemplateCompliance there.
   A regen loop with MAX_REGEN_ATTEMPTS=2 is enforced in
   routes/generation.
   Campaign generation: POST /api/campaigns/:id/generate produces
   campaign_assets rows; QC at GET /api/campaigns/:id/qc (lib/
   campaignQC.ts validates per-channel constraints); activate at
   PATCH /api/campaigns/:id/activate.

H. WORK QUEUE (structural)
   POST /api/work-queue/start creates a work_queue_sessions row, then
   in background creates work_queue_findings (auto_fix or decision_card).
   GET /api/work-queue/cards lists pending decision cards;
   POST /api/work-queue/auto-fix applies all auto-fix findings;
   per-card POST .../accept and .../skip resolve individual ones.
   GET /api/work-queue/summary returns counts; the session row's
   status field tracks RUNNING / READY / COMPLETE / FAILED.

I. DOCUMENT HEALTH (structural)
   POST /api/document-health/run creates a document_health_sessions row
   and writes a document_health_scores row per CURRENT document with
   7 dimension statuses (identity/targeting/belief/compliance/
   propagation/content/delivery). GET /api/document-health/latest
   returns the most recent session+scores for the
   /document-health frontend page.

J. LIVE QUEUE UPDATES (read end-to-end)
   GET /api/events/queue is an SSE stream. lib/queueEvents.ts maintains
   an in-memory subscriber set; webhook handlers and the
   untagged-conversation sweep call notifyQueueChanged() to flush a
   refresh ping to all connected Call Command pages. Single-process —
   no Redis or external pub/sub.

K. UNTAGGED CONVERSATION SWEEP (read end-to-end at orchestration level)
   src/index.ts calls startUntaggedSweep() at boot (exported from
   routes/aircall/index.ts). Comment in app boot logs "Untagged-
   conversation sweep scheduled (every 5 min)". Body of the sweep
   handler not read.

L. COMPLIANCE CONSTANTS (structural)
   Two-step propose/confirm workflow:
     POST /api/compliance-constants/propose →
     POST /api/compliance-constants/confirm.
   src/index.ts loads the cache at boot via
   lib/complianceConstantsService.loadConstants() and refuses to start
   if zero rows are returned (FATAL).

M. DISPLAY LAYER (read end-to-end at routing level)
   The unlock-intel SPA mounts every authenticated page under
   <ProtectedRoute><AppLayout>; admin-only pages additionally wrap in
   <AdminOnlyRoute>. All API calls go through apiClient.apiFetch which
   sends cookies, parses 401 → redirect to /login, and returns Response
   for everything else. React Query is the data-fetching cache for
   most reads; the body of each page (and its specific endpoints)
   was not read in this scan.

SECTION 7 — PATTERNS AND CONVENTIONS
===========================================================================

Project layout & tooling:
  - pnpm workspace monorepo. Three artifacts under artifacts/, shared
    libraries under lib/, helper scripts under scripts/.
  - All TypeScript packages use ESM ("type": "module"). The api-server
    boots from a single esbuild-bundled dist/index.mjs.
  - Per-artifact `typecheck` script runs `tsc --noEmit`; root
    `typecheck` first builds lib project references then fans out
    across artifacts and scripts.
  - npm packages are gated behind a 24-hour minimum release age
    (pnpm-workspace.yaml), with an exception for @replit/* and a few
    pinned trusted packages.
  - Native binaries for non-linux-x64 platforms are explicitly removed
    via overrides (esbuild, lightningcss, @tailwindcss/oxide, rollup,
    @expo/ngrok-bin) — Replit container is linux-x64 only.

Database / ORM:
  - Drizzle ORM with drizzle-zod. Each schema file follows the same
    template: pgTable definition, createInsertSchema(...).omit(...),
    `export type InsertX = z.infer<...>` and `export type X = ...
    $inferSelect`.
  - All "id" columns are text primary keys with a randomUUID-based
    `$defaultFn` and a domain-specific prefix (e.g. `contact_<uuid8>`,
    `run_<uuid8>`, `rev_<uuid12>`, `dec_<uuid12>`, `eis_<uuid8>`,
    `art_<uuid8>`, `mem_<uuid8>`, `calllist_<uuid8>`, `sig_<uuid8>`,
    `trn_<uuid8>`, `upload_<uuid8>`).
  - Timestamps almost universally use `{ withTimezone: true }` and
    .notNull().defaultNow(); updated_at columns use `.$onUpdate(() =>
    new Date())`.
  - JSONB columns default to `[]` or `{}` and are typed via
    drizzle-orm's jsonb helper; richer typing is occasionally added
    via `.$type<...>()`.
  - Foreign keys reference table objects directly (so Drizzle picks up
    the relation), with explicit `onDelete: "set null"` where
    appropriate (e.g. agents.user_id → users).
  - Append-only history tables exist alongside their state tables
    (engine_signals + engine_signal_transitions, lead_beliefs +
    belief_transitions, changelog).
  - Partial unique indexes are used for "at most one active per X"
    invariants (call_list_memberships active per contact).

API server:
  - Express 5 with a single barrel router under /api. Each sub-router
    is `Router()` from express, exported as default; routes/index.ts
    mounts all of them.
  - `pinoHttp` for request logging with custom serialisers; pino
    everywhere for structured logs.
  - Cookies-based session auth via express-session + connect-pg-simple.
    Cookie name `cu.sid`, 7-day rolling expiry, sameSite=lax, secure
    only in production.
  - Auth middleware pair: requireAuth → requireAdmin (composes the
    former). Routers mostly leave gating implicit (most endpoints
    require auth simply by virtue of frontend guarding); requireAdmin
    is applied per-route on settings/admin endpoints.
  - Validation: zod (`zod/v4` for drizzle-zod outputs, `zod` for
    api-spec-derived schemas). Health route demonstrates the pattern of
    parsing the response shape through an api-zod schema before sending.
  - SSE for live updates is implemented with a single in-process
    subscriber set in lib/queueEvents.ts.
  - Background work is started in src/index.ts by calling
    startUntaggedSweep() after boot — there is no external job runner.

Engine layout (engine/v2/, contains V3 implementation):
  - Each engine function lives in its own file under functions/ with a
    leading "C<n>. <name>" comment matching the spec numbering.
  - Functions return strict typed shapes (types.ts is the single source
    of truth) — V3 outputs are documented as "strict superset of V2"
    so existing callers keep working.
  - LLM extraction is a sub-layer (engine/v2/llm/) with a separate
    extraction schema and adapter functions that translate the
    LLMExtractionResult into Layer-2 inputs.
  - Outcome-rule engine is independent (engine/v2/outcomeRules/) and
    is gated behind ENGINE_OUTCOME_RULES env flag (per the
    engine_outcome_rules schema header).
  - Persistence is a separate persistence.ts file; the function layer
    never imports the database directly. Tests/harnesses use
    blankInvestor() fixtures and never touch the DB.

Frontend SPA (unlock-intel):
  - Routing via wouter. Single Switch in App.tsx, wrapped in
    QueryClientProvider, TooltipProvider, and a base-path WouterRouter
    derived from import.meta.env.BASE_URL.
  - Two gating components: ProtectedRoute (every page except /login)
    and AdminOnlyRoute (admin-only pages). Both rely on a single
    useCurrentUser query hook with staleTime 60s and retry:false.
  - All API calls go through apiClient.apiFetch which always sends
    `credentials: "include"` and on a 401 forces a top-level navigation
    to /login?returnTo=<path> (no inline auth UI).
  - shadcn/ui primitives sit under components/ui/ — ~50 small wrappers
    around Radix UI; the same set is replicated almost verbatim under
    artifacts/mockup-sandbox/src/components/ui/.
  - Tailwind v4 with @tailwindcss/vite. Theming via CSS variables in
    src/index.css (file presence confirmed; not opened).
  - React Query is the data-fetching cache; non-React-Query useEffect
    fetches are also used in places.

Code style:
  - File-leading block comments are used heavily, particularly in
    engine functions and in schema files where they describe the
    column's role and the operational semantics around it.
  - Comments often reference phase numbers (Phase 4.7, 4.9, 7.1a/b,
    010c/d, 011a-c, 012a/c, 013a/b/c) tying code to the changelog in
    replit.md.
  - Constants and shared types live in lib/ (lib/personas.ts,
    lib/coverage-matrix.ts, lib/call-questions.ts) so both server and
    client can import them.
  - api-zod and api-client-react are produced by Orval from a single
    openapi.yaml, treating the API spec as the contract and generating
    both sides.


===========================================================================
SECTION 8 — INCONSISTENCIES AND FLAGS
===========================================================================

Observations only. No fixes are proposed here.

  1. Engine version naming.
     The directory is artifacts/api-server/src/engine/v2/ but
     ENGINE_VERSION = "3.0.0" and ENGINE_SPEC = "402_SPEC_Intelligence_
     Engine_V3_CURRENT.md". The barrel file is labelled "V2 intelligence
     engine" in routes/engine/index.ts's leading comment, while
     processTranscript.ts says "V3 orchestrator. Supersedes V2's C10."
     Path name has not caught up with the version it now contains.

  2. Engine spec file in docs/ does not match the spec the code claims.
     docs/402_SPEC_Intelligence_Engine_V2_CURRENT.md is checked in;
     engine/v2/version.ts references
     402_SPEC_Intelligence_Engine_V3_CURRENT.md, which does not appear
     under docs/.

  3. Frontend role union narrower than backend.
     useCurrentUser.ts declares `role: "agent" | "admin"`. The users
     schema and assigned_closer_id semantics on contacts establish a
     third role "closer" used by the backend (see comment block in
     users schema). A closer login would land in CurrentUser with a
     role value the frontend type doesn't enumerate.

  4. Schema comment vs actual provider key for Google.
     integration_configs.ts comments document a `google_calendar`
     provider config (client_id/secret/redirect_uri/access_token/
     refresh_token), but the same OAuth credentials are described by
     replit.md as being used for the Google Docs / Drive export
     pipeline. The provider key name and the de-facto usage are not
     fully aligned.

  5. Two parallel "conversation" tables.
     `conversations` (serial id, plus messages with ON DELETE CASCADE)
     looks generic / scaffolded, while `lead_conversations` (text id,
     transcript_text, beliefs_before/after, etc.) is the domain
     conversation table. Whether `conversations` is still used
     anywhere is not visible from this scan; nothing in the route
     mounts referenced it directly.

  6. Two outcome routing layers.
     engine_outcome_rules has an `actions` jsonb column AND legacy
     single-action columns (action_type, owner, timing, detail,
     uses_content). The schema comment says "Loader + evaluator
     prefer `actions` when present"; both shapes are still permitted
     in storage. Migration window is acknowledged in the comment.

  7. determineNextAction.ts and the outcome-rules engine coexist.
     The legacy hardcoded determineNextAction is still exported from
     engine/v2/index.ts and is gated against the ENGINE_OUTCOME_RULES
     env flag at runtime. Two implementations of the same decision
     are kept in tree.

  8. integration_configs.provider value `aircall` vs the enabled
     boolean column.
     The provider column is a free-text unique key (not an enum), and
     the only documented values come from the trailing comment. There
     is no DB-level CHECK constraint enforcing the provider list
     (aircall|pipedrive|fireflies|google_calendar). Out-of-band
     additions would be silently accepted.

  9. lib/integrations/anthropic_ai_integrations is an empty directory.
     The actual Anthropic integration package lives at
     lib/integrations-anthropic-ai (note hyphenation difference). The
     empty directory is still listed under
     pnpm-workspace.yaml's `packages: [..., lib/integrations/*, ...]`
     glob — a future package added under that path would be picked up
     automatically, but right now nothing else is there.

 10. Soft foreign-keys without database-level constraints.
     contacts.lead_id, leads.contact_id, engine_signal_transitions.
     engine_run_id, engine_artifacts_sent.engine_run_id, and
     call_list_memberships.carried_from_id are all documented in
     comments as "soft refs" — they are plain text columns without a
     references() clause, so referential integrity for these links is
     enforced (or not) only by application logic.

 11. work_queue_findings.task_id and document_id are NOT NULL but
     lack FK constraints (per the file: "task_id/document_id no FK").
     Whatever enforcement exists is at the application layer — there
     is no DB-level guarantee that the referenced rows exist.

 12. Two DOM "use-mobile" / "use-toast" hooks.
     Both unlock-intel and mockup-sandbox carry their own
     hooks/use-mobile.tsx and hooks/use-toast.ts copies of what is
     usually a shared shadcn helper. Same files are also duplicated
     across the two artifacts under components/ui/. Drift between the
     two copies is possible.

 13. The deployment manifest in .replit lists artifacts/api-server and
     artifacts/mockup-sandbox but NOT artifacts/unlock-intel, even
     though unlock-intel is the user-facing SPA. unlock-intel does
     appear in the artifacts registry comment block at the top of the
     project description, so this could be intentional (different
     deployment topology), but it is asymmetric with the other two
     artifacts.

 14. Boot-time fatal vs warning behaviour is not symmetric.
     src/index.ts treats compliance_constants being empty as FATAL
     (process.exit(1)). It treats seedDatabase() failing as a warning
     (logs and still calls app.listen). Similarly, app.ts treats a
     missing SESSION_SECRET as a warning (only /api/auth/* fails)
     rather than a hard exit.

 15. Anthropic client throws at module load if env vars are missing,
     but it is imported transitively by routes/recommendation/index.ts
     and routes/aircall/index.ts. Booting api-server without those
     env vars would fail the import resolution for the whole app, not
     just the routes that actually need Claude.

 16. The "sheet-sync" sub-router is the only one mounted under a
     prefix path (`router.use("/sheet-sync", sheetSyncRouter)`); the
     other 34 sub-routers either declare the full `/api/...` path
     inside their handlers or have no prefix. This makes
     /api/sheet-sync routes the only ones whose mounted prefix is not
     visible at the handler.
