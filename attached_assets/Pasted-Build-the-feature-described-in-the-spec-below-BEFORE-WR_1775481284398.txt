## Section 1 — Feature

Adds the Work Queue page at `/work-queue`. The operator presses one button to start a compliance audit. The system analyses all open Review tasks in the background. When ready, it shows a Phase 1 summary (auto-fixes applied silently) and then a Phase 2 card flow — one decision card at a time, Accept or Skip, until the queue is empty. Ends with a session summary. A sidebar badge shows the open task count.

---

## Section 2 — Why this is necessary

55 compliance findings exist in the DB with no UI to see or act on them. 7 fully working backend endpoints have no frontend. The entire Work Queue system is invisible to the operator.

### 5.2 — Sidebar badge

File: wherever the sidebar navigation is defined (find it — likely a layout or sidebar component).

Add a "Work Queue" nav item with a live task count badge:

```tsx
// Fetch open Review task count on sidebar mount
const [openTaskCount, setOpenTaskCount] = useState<number>(0);

useEffect(() => {
  const fetchCount = async () => {
    try {
      const res = await fetch(`${API_BASE}/tasks?status=Open&type=Review`);
      const data = await res.json();
      setOpenTaskCount(data.tasks?.length || 0);
    } catch { /* silent */ }
  };
  fetchCount();
}, []);
```

Show badge only when count > 0:
```tsx
<NavItem href="/work-queue" icon={Zap}>
  Work Queue
  {openTaskCount > 0 && (
    <Badge variant="destructive" className="ml-auto text-xs">
      {openTaskCount}
    </Badge>
  )}
</NavItem>
```

### 5.3 — Work Queue page

File: `artifacts/unlock-intel/src/pages/work-queue.tsx` (create new file)

The page has 5 states. The UI shows exactly one state at a time:

```
IDLE → PREPARING → PHASE_1_SUMMARY → PHASE_2_CARDS → SUMMARY
```

**State machine:**
```typescript
type QueueState = 'IDLE' | 'PREPARING' | 'PHASE_1_SUMMARY' | 'PHASE_2_CARDS' | 'SUMMARY';
```

---

### 5.4 — IDLE state

Show when: no active session exists (GET /status returns session: null) AND no READY/COMPLETE session.

Also show on first load before status is checked.

UI:
```
[Large centered card]
  Icon: Zap (large, accent colour)
  Heading: "Work Queue"
  Subtext: "Run a compliance audit across all open review tasks.
            The system will automatically fix objective violations
            and surface content decisions for your review."

  [Start button — primary, large]
  Label: "Start Compliance Audit"
  On click: call POST /work-queue/start → transition to PREPARING

  [Small note below button]
  "This will analyse all open Review tasks against compliance
   constants and document content."
```

If GET /status returns a READY session on page load: skip IDLE, go straight to PHASE_1_SUMMARY.
If GET /status returns ANALYSING: skip to PREPARING.

---

### 5.5 — PREPARING state

Show when: session status is PENDING or ANALYSING.

Poll GET /status every 2 seconds. When status becomes READY: transition to PHASE_1_SUMMARY.

UI:
```
[Centered card]
  Icon: Loader2 (spinning, accent colour)
  Heading: "Analysing documents..."
  Subtext: "Claude is reviewing your documents against compliance
            rules. This takes about 30 seconds."

  [Progress bar]
  Progress: (session.analysed_tasks / session.total_tasks) * 100
  Label: "Analysed {analysed_tasks} of {total_tasks} documents"

  [Small note]
  "You can leave this page — the analysis continues in the background."
```

Stop polling when status = READY, COMPLETE, or FAILED.
On FAILED: show error state with session.error_message and a "Try Again" button.

---

### 5.6 — PHASE_1_SUMMARY state

Show when: session status is READY. Show BEFORE cards.

First: automatically call POST /work-queue/auto-fix. Show a loading spinner while this runs.

After auto-fix completes, show:

```
[Centered card — green tint if applied > 0, neutral if 0]

  Icon: CheckCircle (green)
  Heading: "Phase 1 Complete"

  [Stats row]
  "N compliance issues fixed automatically"
  "N documents updated"
  "N downstream documents flagged for review"

  [If applied === 0:]
  "No automatic fixes needed — all compliance figures are correct."

  [If session.cards_total > 0:]
  [Continue button — primary]
  Label: "Review {cards_total} content decisions →"
  On click: transition to PHASE_2_CARDS

  [If session.cards_total === 0:]
  [Done button — primary]
  Label: "All done — queue is clear"
  On click: transition to SUMMARY
```

---

### 5.7 — PHASE_2_CARDS state

Show when: cards remain (GET /cards returns cards.length > 0).

On entering this state: fetch GET /work-queue/cards to load all pending cards.
Show cards one at a time — the first card in the sorted array (sort_order ascending).

**Card UI:**
```
[Full-width card — clean, focused]

  [Top bar]
  Left: Document tier badge ("Tier 1" / "Tier 2" / "Tier 3")
        Document name in bold
  Right: Progress indicator: "Card 3 of 12"

  [Issue section]
  Label: "Issue identified"
  Content: finding.issue_description

  [If original_text set:]
  Label: "Current text"
  Content: <code block> finding.original_text </code>

  [If proposed_fix set:]
  Label: "Suggested fix"
  Content: <code block or styled diff> finding.proposed_fix </code>

  [Action row — two buttons]
  [Accept button — green, primary]
    Icon: CheckCircle
    Label: "Accept fix"
    On click: call POST /cards/:id/accept → remove card from local array → show next card
    Show Loader2 while in-flight

  [Skip button — ghost/outline]
    Icon: XCircle
    Label: "Skip"
    On click: show skip reason selector (inline, below the button row)

  [Skip reason selector — appears inline when Skip is clicked]
    Three buttons (not a dropdown):
    "Not relevant" | "Needs more context" | "Defer"
    On selection: call POST /cards/:id/skip with { reason } → remove card → show next
```

**Card transitions:**
- After accept or skip: remove the current card from local state, show the next card
- No animation required — just swap the content
- After the last card: call GET /work-queue/summary → transition to SUMMARY

**Progress tracking:**
- Track `currentIndex` locally (starts at 0, increments after each accept/skip)
- Show "Card {currentIndex + 1} of {cards.length}"

---

### 5.8 — SUMMARY state

Show when: all cards have been resolved OR session.status === COMPLETE.

UI:
```
[Centered card]
  Icon: CheckCircle (large, green)
  Heading: "Queue complete"
  Subtext: "Here's what happened in this session."

  [Stats grid — 2×2 or 1×4]
  "Auto-fixed"     {summary.auto_fixed}
  "Accepted"       {summary.cards_accepted}
  "Skipped"        {summary.cards_skipped}
  "Cascaded"       {summary.cascaded}

  [If summary.still_open > 0:]
  Amber note: "{still_open} findings were skipped and remain open.
               They will appear in the next queue run."

  [Two buttons]
  [Run again — secondary]
  On click: call POST /start → back to PREPARING

  [Back to tasks — ghost]
  On click: navigate to /tasks
```

---

### 5.9 — Error handling

- If POST /start returns `already_running: true`: do not transition to PREPARING — instead show a toast/note "Analysis already running" and poll GET /status.
- If any card action (accept/skip) fails: show an error message below the card, do not remove the card. Allow retry.
- If GET /cards returns empty unexpectedly: go straight to SUMMARY.
