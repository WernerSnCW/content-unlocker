## Section 1 — Feature

This build migrates the 19 compliance constants currently stored in
`compliance_constants.json` into a PostgreSQL table managed via Drizzle ORM,
introduces a governed lifecycle (ACTIVE / DRAFT / SUPERSEDED) with a two-step
override-confirmation flow, and exposes the constants through API endpoints and
a minimal admin UI. It replaces the startup JSON load in `dataManager.ts` with
a service/repository layer that caches constants in memory and invalidates on
update. This is Phase A of the import pipeline project — it makes compliance
constants writable at runtime through a safe, auditable path, which Phase B
(the full import pipeline) depends on.

---

## Section 2 — Why this is necessary

Currently, compliance constants live in `compliance_constants.json` and are
loaded at startup via a JSON import assertion in `dataManager.ts`. There is no
API endpoint, no lifecycle control, no audit trail, and no way to update a
constant value without a code/config change and server restart. The 19-check QC
engine in `generation/index.ts` references these constants for the
TAX_FIGURE_ACCURACY check — an incorrect value silently corrupts every document
QC'd after it. The Phase B import pipeline cannot safely write to compliance
constants without a governed update path. Without this phase, updating a
compliance figure (e.g. the VCT rate from 30% to 20%) requires a deployment
rather than an auditable runtime operation.

### 5.3 Service layer

Create `artifacts/api-server/src/lib/complianceConstantsService.ts`:

**In-memory cache:**
```typescript
let _cache: ComplianceConstant[] | null = null;
```

**`loadConstants()`** — fetches all status=ACTIVE records from DB, populates
cache. Called at startup and after any successful override confirmation.

**`getConstants()`** — checks feature flag `compliance_constants_db` first.
If flag is false: loads from `compliance_constants.json` directly and returns
without DB access. If flag is true: returns cache. If cache is null, calls
loadConstants() first. Never returns null — throws if DB is unreachable.

**`getConstantByKey(key: string)`** — returns single constant from cache by key.
Returns undefined if not found (caller handles missing key).

**`invalidateCache()`** — sets `_cache = null`. Called by override confirmation
route after successful write.

**`validateOverride(key: string, newValue: string)`** — checks:
- Key exists and has status=ACTIVE
- New value is not a prohibited value (22p, 7.8x, 78x) regardless of which
  constant is being updated — if new value matches any prohibited string, reject
- If constant has `subject_to_qualifier: true`, new value must not remove the
  qualifier requirement
- Returns `{ valid: boolean, error?: string }`

### 5.5 API routes

Create `artifacts/api-server/src/routes/compliance-constants/index.ts`.

Mount in `artifacts/api-server/src/routes/index.ts` as `/compliance-constants`.

**Static routes first, then parameterized:**

`GET /compliance-constants`
- Returns all ACTIVE constants from cache (via getConstants())
- Response: `{ constants: ComplianceConstant[] }`

`GET /compliance-constants/categories`
- Returns list of distinct category values from ACTIVE constants
- Static route — must be declared before /:id

`POST /compliance-constants/propose`
- Creates a DRAFT record for a proposed override
- Body: `{ key, newValue, reason, actor }`
- Validates: key exists as ACTIVE, newValue passes validateOverride()
- If prohibited value detected: returns 400 with `{ error: "Prohibited value", prohibited_value: newValue }`
- If subject_to_qualifier constant and reason is absent: returns 400
- On success: inserts DRAFT record, returns `{ draft_id, key, currentValue, proposedValue, reason }`
- Does NOT activate the override — confirmation required

`POST /compliance-constants/confirm`
- Confirms and activates a proposed DRAFT override
- Body: `{ draft_id, confirmation_text }` — confirmation_text must be non-empty
- Marks current ACTIVE record as SUPERSEDED (sets superseded_by = new record id)
- Marks DRAFT record as ACTIVE, sets activated_at = now()
- Calls invalidateCache() then loadConstants()
- Flags any documents in documentsTable where content references the changed
  constant key as REQUIRES_REVIEW (query documents where content ILIKE
  '%' || old_value || '%' — set review_state = 'REQUIRES_REVIEW')
- Creates changelog entry: `{ document_id: null, action: 'COMPLIANCE_CONSTANT_UPDATED', details: { key, old_value, new_value, reason, actor, source } }`
- Returns `{ success: true, activated_id, superseded_id, documents_flagged: number }`

`GET /compliance-constants/key/:key`
- Returns full history for a key: ACTIVE record + all SUPERSEDED records ordered by activated_at desc
- Declare after /categories, before /:id

`GET /compliance-constants/:id`
- Returns single constant record by id (any status)
- Parameterized — declared after all static routes

## Section 10 — Rollback

Risk is HIGH — rollback plan required.

1. **Feature flag**: set `compliance_constants_db: false` in environment —
   complianceConstantsService falls back to JSON load. No data loss.
2. **Schema rollback**: compliance_constants table can be dropped without
   affecting any other table (no foreign keys from existing tables point to it).
   Run: `DROP TABLE compliance_constants;`
3. **File rollback**: revert any changes to generation/index.ts, dataManager.ts,
   layout.tsx, routes/index.ts, App.tsx. generationEngine.ts is unchanged.
4. **External state**: documents flagged as REQUIRES_REVIEW can remain — this
   is a safe state. No documents are deleted or content-modified by this build.
5. **Cannot roll back**: changelog entries written during constant overrides
   (if any were performed before rollback). These are append-only and harmless.
