# BUILD 055 — Multi-Transcript Batch Upload
Priority: HIGH
Area: Recommendation Engine — Analyze

## Overview

Users need to upload multiple call transcripts in one session.
Each transcript is analyzed independently — producing its own
persona, pipeline stage, and document recommendations. After
reviewing results, the user decides which lead to link each
transcript to (existing or new).

---

## Step 1 — Backend: File parsing endpoint

Create a new endpoint: POST /api/recommendation/parse-transcripts

This accepts multipart/form-data with one or more uploaded files.

Supported formats:
- .txt — read as plain text (UTF-8)
- .docx — extract plain text from the document body only
  (ignore headers, footers, comments)

Response body:
```typescript
{
  parsed: Array<{
    filename: string
    content: string
    error?: string    // present if file could not be parsed
  }>
}
```

Rules:
- Accept up to 20 files per request — reject with 400 if exceeded
- Enforce a 500KB size limit per file — reject oversized files with
  a clear per-file error, do not reject the whole batch
- If a file cannot be parsed (wrong format, corrupt, empty), include
  it in the response with an error field rather than failing the
  whole request
- Return the parsed content in the same order as the uploaded files

---

## Step 2 — Backend: Batch analyze endpoint

Create a new endpoint: POST /api/recommendation/analyze-batch

Request body:
```typescript
{
  transcripts: Array<{
    filename: string        // original filename, used as display label
    content: string         // plain text content of the transcript
  }>
}
```

Response body:
```typescript
{
  results: Array<{
    filename: string
    status: "success" | "error"
    analysis?: {
      persona: string
      persona_confidence: number
      stage: string
      stage_confidence: number
      objections: string[]
      evidence: string[]
    }
    error?: string          // present only if status is "error"
  }>
}
```

Processing rules:
- Process each transcript sequentially, not in parallel — Claude
  calls must not be concurrent to avoid rate limit errors
- If one transcript fails, continue processing the rest — do not
  abort the batch
- Use exactly the same Claude prompt and persona guide context as
  the existing single analyze endpoint
- Do not create leads, log sends, or write to the database
- The batch endpoint is analysis only — lead linking happens in
  the frontend after the user reviews results

---

## Step 3 — Frontend: Batch upload UI

Add a "Batch Upload" mode to the existing Recommend page,
accessible via a toggle or tab alongside the existing single
transcript input.

**Upload area:**
- Drag-and-drop zone accepting .txt and .docx files
- Also accepts click-to-browse file selection
- Show each uploaded file as a removable chip/tag before submission
- Display file count and total size as files are added
- Show a clear error if more than 20 files are selected
- Show per-file errors for files that exceed 500KB

**Processing state:**
- On submit, call POST /api/recommendation/parse-transcripts first
- Then call POST /api/recommendation/analyze-batch with the parsed
  content
- Show a progress indicator: "Analyzing transcript X of Y" updating
  as each result arrives
- Do not show a spinner for the whole batch — show incremental
  progress

**Results view:**
After all transcripts are analyzed, display results as a list.
Each result card shows:
- Filename as the card title
- Detected persona and confidence score
- Detected pipeline stage and confidence score
- Key objections detected (up to 3, truncated)
- A status badge: ANALYZED / ERROR

If a transcript errored, show the error message in the card and
allow the user to remove it or re-upload.

**Lead linking (per transcript):**
Each result card includes a "Link to Lead" action. On click, show
a dropdown or modal with two options:
- "Match existing lead" — searchable dropdown of current leads
  (search by name or company)
- "Create new lead" — opens the standard new lead form
  pre-populated with any name/company detected in the transcript

Linking a transcript to a lead navigates to the standard single
Recommend flow for that lead, with the analysis result
pre-populated (persona, stage, objections already filled in —
user proceeds directly to the Rank step without re-analyzing).

Do not auto-link or auto-create leads — the user must make this
decision explicitly for each transcript.

## Step 2 — Backend: Enhanced analyze endpoint

Update the existing POST /api/recommendation/analyze endpoint.
The enhancement is additive — existing integrations must continue
to work unchanged.

Add the following to the existing Claude prompt:

```
In addition to investor persona and pipeline stage, assess the
following.

When detecting pipeline stage, use only these canonical values:
Outreach | Called | Demo Booked | Demo Complete | Decision

READINESS TO CLOSE
Buying signals include:
- Asking about process, next steps, or paperwork
- Mentioning a specific investment amount
- Referencing tax year deadlines or timing
- Comparing the product favourably to alternatives
- No unresolved objections remaining
Rate readiness as: READY | WARM | NURTURING | COLD
Provide specific evidence from the transcript for the rating.

BLOCKING OBJECTIONS
Objections that are actively preventing progress include:
- Needing to speak to an accountant, IFA, or partner before deciding
- Risk or downside concerns expressed more than once
- Uncertainty about product structure or mechanics
- Hesitation language: "not sure", "need to think", "maybe later"
- Timeline deflection without a specific date
For each: name it, quote the evidence, and rate severity as
CRITICAL | SIGNIFICANT | MINOR.

INFORMATION GAPS
Topics the investor is confused about or expressing misconceptions
about include:
- Direct questions about how EIS, SEIS, or VCT works
- Incorrect comparisons to ISAs, pensions, or standard equities
- Confusion about tax relief timing or mechanics
- Misunderstanding of risk or loss relief
For each: name it, describe the confusion, and identify what
document or explanation would address it.

PRIMARY ISSUE
The single most important thing driving next action. Choose one:
- READY_TO_CLOSE — buying signals present, objections resolved
- OBJECTION_TO_RESOLVE — a specific blocking objection must be
  addressed first
- INFORMATION_GAP — investor needs specific education before
  they can decide
- NEEDS_NURTURING — too early, insufficient signals to act on yet
Provide a one-sentence recommended next action for the salesperson.
```

Add to the analyze response:
```typescript
{
  // existing fields unchanged
  persona: string
  persona_confidence: number
  stage: string                    // must be one of the 5 canonical stages
  stage_confidence: number
  objections: string[]
  evidence: string[]

  // new fields
  readiness: "READY" | "WARM" | "NURTURING" | "COLD"
  readiness_evidence: string[]
  blocking_objections: Array<{
    name: string
    evidence: string
    severity: "CRITICAL" | "SIGNIFICANT" | "MINOR"
  }>
  information_gaps: Array<{
    topic: string
    description: string
    suggested_document_type: string
  }>
  primary_issue: "READY_TO_CLOSE" | "OBJECTION_TO_RESOLVE" | "INFORMATION_GAP" | "NEEDS_NURTURING"
  recommended_next_action: string
}
```

If the Claude response returns a stage value outside the canonical
list, map it to the closest canonical stage or return
stage_confidence: 0 with a note. Never pass a non-canonical stage
value to the frontend.

# BUILD 059 — Gap Analysis Persistence
Priority: MEDIUM
Area: Content Gap Analysis
Depends on: Build A (056 v1.1) must be complete — confirm before
starting.

## Overview

Gap analysis results currently exist only for the duration of a
session. This build adds three persistence layers:
1. Database — every gap analysis run saved as a snapshot record
2. Project file — each run writes a dated JSON file to the
   project directory
3. Frontend export — users can download any saved snapshot as
   JSON or markdown

If one layer fails, the others must still complete successfully.

---

## Step 1 — Database: Gap snapshot schema

Add a new table: gap_snapshots

```typescript
gap_snapshots {
  id:              string        // e.g. "gap_20260403_143022"
  created_at:      timestamp
  matrix_gaps:     jsonb
  type_gaps:       jsonb
  recommendation_gaps: jsonb
  information_readiness: jsonb
  summary:         jsonb
  total_gaps:      integer       // denormalised from summary
  file_path:       string        // path of the project file written
  notes:           string        // optional user label
}
```

Add two endpoints:

GET /api/content/gaps/history
Returns all snapshots in reverse chronological order:
```typescript
{
  snapshots: Array<{
    id: string
    created_at: string
    total_gaps: number
    matrix_gap_count: number
    type_gap_count: number
    recommendation_failure_count: number
    file_path: string
    notes: string
  }>
}
```

GET /api/content/gaps/history/:snapshotId
Returns the full snapshot record including all gap arrays.

---

## Step 2 — Backend: Auto-save on every gap run

Update GET /api/content/gaps to automatically save a snapshot
after every successful run.

Snapshot ID format: gap_{YYYYMMDD}_{HHmmss}

Add to the existing response:
```typescript
{
  // ... all existing fields ...
  snapshot_id: string
  snapshot_file: string    // relative path of the written project file
}
```

If the database save fails: log the error, still return gap
results — do not block the user.
If the file write fails: log the error, still return results and
the snapshot_id — do not block.

---

## Step 3 — Project file: Write dated JSON snapshot

After each gap run, write a JSON file to:
reports/gap-analysis/{snapshot_id}.json

Create the directory if it does not exist.

File contents: the complete gap analysis response as JSON.

Also maintain a manifest at reports/gap-analysis/manifest.json
listing all snapshots in reverse chronological order:
```json
{
  "snapshots": [
    {
      "id": "gap_20260403_143022",
      "created_at": "2026-04-03T14:30:22Z",
      "total_gaps": 0,
      "file": "reports/gap-analysis/gap_20260403_143022.json"
    }
  ]
}
```

Update the manifest on every new run. If the manifest does not
exist, create it. If it is malformed, recreate it from the files
present in the directory.

---

## Step 4 — Backend: Export endpoints

GET /api/content/gaps/history/:snapshotId/export?format=json
Response headers:
- Content-Type: application/json
- Content-Disposition: attachment; filename="gap-analysis-{id}.json"

GET /api/content/gaps/history/:snapshotId/export?format=markdown

- Content-Disposition: attachment; filename="gap-analysis-{id}.md"

```
[rows]

## Document Type Gaps

[rows]

|---|---|---|
[rows]
```
