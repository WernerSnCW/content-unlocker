## Fix 1 — Add standalone /recommendation/gap-brief endpoint
**Priority:** MEDIUM — broken flow for agents mid-call
**File:** recommendation/index.ts

### The problem
When the rank endpoint detects a content gap, the gap panel in
the recommend flow shows a "View Content Gap Analysis" button
that navigates to /gaps. This breaks the agent's context — they
have to leave the recommend flow, navigate to a different page,
and find the right gap manually.

The spec called for POST /api/recommendation/gap-brief — a
dedicated endpoint that generates a brief inline within the
recommend flow without navigation.

### The fix

Create a new endpoint: POST /api/recommendation/gap-brief

This wraps the existing POST /api/content/generate-brief endpoint
internally, passing the recommendation context as the gap source.

Request body:
```typescript
{
  archetype: string          // from the failed recommendation
  stage: string              // canonical stage value
  persona: string            // detected persona (more specific
                             // than archetype)
  transcript_summary: string // context from the current call
}
```

Implementation:
- Construct a gap object in the format expected by generate-brief:
  ```typescript
  {
    archetype: archetype,
    stage: stage,
    gap_type: "recommendation_failure"
  }
  ```
- Pass transcript_summary as additional_context alongside the
  master generation context (which auto-loads)
- Call the generate-brief logic internally (not via HTTP — import
  and call the function directly to avoid internal HTTP overhead)
- Return the brief in the same shape as POST /api/content/generate-brief

Response body (same as generate-brief):
```typescript
{
  brief: {
    title: string
    document_type: string
    archetypes: string[]
    stages: string[]
    key_messages: string[]
    tone: string
    length_guidance: string
    compliance_considerations: string[]
    source_material_pointers: string[]
    information_needed: Array<{
      field: string
      description: string
      source: string
    }>
  }
  ready_to_generate: boolean
}
```

### Update the gap panel UI

Update the gap panel in recommend.tsx to use the new endpoint.

Replace the "View Content Gap Analysis" button with a
"Generate Brief" button that:
1. Calls POST /api/recommendation/gap-brief with the current

Keep the "View Content Gap Analysis" link as a secondary option
below the "Generate Brief" button — do not remove it.

## Fix 2 — Add relevance_score to rank prompt response schema
**Priority:** MEDIUM — low-relevance gap detection may never trigger
**File:** recommendation/index.ts — the rank endpoint Claude prompt

### The problem
The rank prompt asks Claude to return document_id, priority, and
rationale. It does not explicitly request relevance_score.

The low-relevance gap detection added in build 078 checks whether
all ranked results have relevance_score < 0.4. If Claude doesn't
return relevance_score (because it's not in the requested schema),
the check will never trigger — the gap goes undetected even when
all documents are weakly matched.

### The fix

Update the rank Claude prompt to explicitly request relevance_score
as a required field in the response schema.

Add to the existing rank prompt's response format instruction:

```
For each document in your ranking, return exactly these fields:
- document_id: string (the document's ID)
- priority: number (1 = highest priority)
- rationale: string (one sentence explaining why this document
  fits this investor at this stage)
- relevance_score: number between 0.0 and 1.0 (how relevant
  this document is to the investor's current situation —
  1.0 = perfect fit, 0.5 = moderate fit, 0.1 = weak fit)

A relevance_score below 0.4 means the document is a poor fit
and should be used with caution. Return relevance_score honestly
— do not inflate it.
```

Do not change any other part of the rank prompt, the endpoint
logic, or the response shape. The relevance_score field is already
handled by the low-relevance detection code — this fix ensures
Claude actually returns it.
