## Section 1 — Feature

Adds a task manager to Unlock Intel. A new tasks table stores tasks with a title, status, type, and optional link to a document. Tasks can be created manually via a Task Board page. The Dashboard gains a summary panel showing the operator's open task count and the three oldest outstanding tasks.

---

## Section 2 — Why this is necessary

The 31-document library import has created a large number of documents with REQUIRES_REVIEW status. Currently there is no mechanism to track, queue, or mark complete any review work. The operator has no visibility into what needs attention or how to work through it systematically. Without this feature, review debt is invisible and unmanageable.

### 5.2 — API: tasks router

File: `artifacts/api-server/src/routes/tasks/index.ts` (create new file)

Create an Express Router. Declare routes in this exact order (static before parameterised):

**GET /tasks**
- Query params: none — return all tasks
- Query all rows from `tasksTable` ordered by `created_at` ascending
- For `linked_document_name`: use a Drizzle left join on `documentsTable` where `documentsTable.id = tasksTable.linked_document_id`, selecting `documentsTable.name` as `linked_document_name`. Alternatively, a secondary `db.select().from(documentsTable).where(eq(documentsTable.id, task.linked_document_id))` lookup per task is also acceptable. Either approach is valid.
- Return: `{ tasks: Task[] }` where each task includes `id`, `title`, `status`, `type`, `linked_document_id`, `linked_document_name` (string or null), `created_at`, `updated_at`
- Error: 500 with `{ error: 'Failed to fetch tasks' }` on DB failure

**POST /tasks**
- Body: `{ title: string, type?: string, linked_document_id?: string }`
- Validate: `title` must be non-empty string. If missing or empty: return 400 `{ error: 'title is required' }`
- `type` defaults to `'General'` if not provided. If provided, must be one of `'Review'`, `'Build'`, `'Import'`, `'General'` — else 400 `{ error: 'Invalid type' }`
- `status` always set to `'Open'` on creation — not accepted from body
- Generate `id` via `randomUUID()` from `crypto`
- Insert into `tasksTable`
- Write changelog entry inline: `db.insert(changelogTable).values({ id: randomUUID(), action: 'TASK_CREATED', details: 'Task created: [title]', triggered_by: 'operator' })`
- Return: 201 with the created task object
- Error: 500 with `{ error: 'Failed to create task' }` on DB failure

**GET /tasks/summary**
- Must be declared BEFORE `GET /tasks/:id` in the file
- Query `tasksTable` for all tasks where `status != 'Done'`
- Return:
  - `open_count`: total count of non-Done tasks
  - `top_tasks`: the 3 oldest non-Done tasks ordered by `created_at` ascending, each including `id`, `title`, `status`, `type`, `linked_document_id`, `linked_document_name`
- Return: `{ open_count: number, top_tasks: Task[] }`
- Error: 500 with `{ error: 'Failed to fetch task summary' }` on DB failure

**PATCH /tasks/:id**
- Body: `{ status?: string, title?: string, type?: string }`
- If `status` provided: must be one of `'Open'`, `'In Progress'`, `'Done'` — else 400 `{ error: 'Invalid status' }`
- If `type` provided: must be one of `'Review'`, `'Build'`, `'Import'`, `'General'` — else 400 `{ error: 'Invalid type' }`
- Free status transitions — no sequence enforcement
- If task not found: 404 `{ error: 'Task not found' }`
- Update the task, return the updated task object
- Write changelog entry inline: `db.insert(changelogTable).values({ id: randomUUID(), action: 'TASK_UPDATED', details: 'Task updated: [title]', triggered_by: 'operator' })`
- Return: 200 with updated task object
- Error: 500 with `{ error: 'Failed to update task' }` on DB failure

**DELETE /tasks/:id**
- If task not found: 404 `{ error: 'Task not found' }`
- Delete the task
- Write changelog entry inline: `db.insert(changelogTable).values({ id: randomUUID(), action: 'TASK_DELETED', details: 'Task deleted: [title]', triggered_by: 'operator' })`
- Return: 200 `{ success: true }`
- Error: 500 with `{ error: 'Failed to delete task' }` on DB failure

File: `artifacts/api-server/src/routes/index.ts` (additive edit)

Import the tasks router and mount it: `router.use('/tasks', tasksRouter)` — add after the existing import route mount.

AC3. `POST /api/tasks` with body `{ "title": "" }` returns 400 with `{ "error": "title is required" }`.

AC4. `POST /api/tasks` with body `{ "title": "Test", "type": "InvalidType" }` returns 400 with `{ "error": "Invalid type" }`.

AC5. `PATCH /api/tasks/:id` with body `{ "status": "Done" }` on an existing task with `status: "Open"` returns 200 with `status: "Done"`. A row exists in the changelog table with `action = 'TASK_UPDATED'`.

AC6. `PATCH /api/tasks/:id` with body `{ "status": "InvalidStatus" }` returns 400 with `{ "error": "Invalid status" }`.

AC7. `PATCH /api/tasks/:id` using a non-existent id returns 404 with `{ "error": "Task not found" }`.

AC8. `DELETE /api/tasks/:id` on an existing task returns 200 `{ "success": true }`. No row with that id exists in the tasks table. A row exists in the changelog table with `action = 'TASK_DELETED'`.

AC9. `GET /api/tasks/summary` returns `{ open_count: N, top_tasks: [...] }` where `open_count` equals the number of tasks in the database with `status != 'Done'`, and `top_tasks` contains at most 3 tasks ordered by `created_at` ascending.

AC10. `GET /api/tasks/summary` returns the summary object and not a 404 or task-by-id response. Manual test required: start the server, call `GET /api/tasks/summary` directly (e.g. via curl or browser), confirm the response is `{ open_count, top_tasks }`.

AC11. The Task Board page at `/tasks` renders without error. Manual test required: navigate to `/tasks` in the browser, confirm the page loads showing the create form, two filter selects, and a task list area.

AC12. Creating a task via the Task Board form with a valid title results in the task appearing in the list without a full page reload. Manual test required: fill in a title, click Add Task, confirm the new task appears in the list and the form clears.

AC13. Submitting the Task Board create form with an empty title shows the inline error "Title is required" and does not call `POST /api/tasks`. Manual test required: leave title blank, click Add Task, confirm the error text appears and no network request is made.

AC14. Changing the status `<select>` on a task row updates the task status and the list reflects the new value after the API call completes. Manual test required: change a task from Open to Done using the row select, confirm the displayed status updates.

AC15. The status filter `<select>` filters the task list client-side without an additional API call. Manual test required: with tasks of at least two different statuses present, select "Done" — confirm only Done tasks are shown.

AC16. The Dashboard at `/` shows a Tasks card with the correct open task count and up to 3 task titles. Manual test required: create 4 tasks with status Open, navigate to `/`, confirm the count displays 4 and exactly 3 task titles are listed (the 3 oldest by created_at).

AC18. All task ids stored in the database are UUIDs in the format `xxxxxxxx-xxxx-xxxx-xxxx-xxxxxxxxxxxx` (36 characters). Manual test required: create a task and inspect the `id` field in the database.

- The `$onUpdate` Drizzle helper is available in this codebase — confirmed from the leads schema which uses the same pattern.
- `API_BASE` environment variable is available in the frontend for manual fetch calls — consistent with CTX architecture notes.

- `CheckSquare` is assumed available in the installed version of lucide-react. If not, `ClipboardList` is the fallback — stated explicitly in Section 5.4.
- The dashboard page can accommodate an additional independent fetch call — task list is expected to be small (under 100 items) and does not require caching.
