# BUILD 075 — Transcript-to-Lead Auto-Creation
Priority: HIGH
Area: Recommendation Engine — Batch Upload

## Overview

When batch analysis extracts an investor name from an Aircall
transcript filename and no matching lead exists in the database,
the user currently has to create the lead manually before linking.
This build closes that gap — offering one-click lead creation
pre-populated with the extracted name, detected persona, and
detected pipeline stage directly from the batch result card.

---

## Step 1 — Backend: Fuzzy lead matching endpoint

Create a new endpoint: GET /api/leads/match?name={name}

Matching logic:
- Exact match (case-insensitive): confidence 1.0
- All words present in any order: confidence 0.85
- First name match only: confidence 0.5
- Last name match only: confidence 0.4
- No match above 0.4: return empty array

Response body:
```typescript
{
  matches: Array<{
    lead_id: string
    name: string
    company: string
    pipeline_stage: string
    detected_persona: string
    confidence: number
  }>
  query: string
}
```

Return up to 3 matches ordered by confidence descending.

---

## Step 2 — Backend: Pre-populated lead creation

Update POST /api/leads to accept optional analysis fields:

```typescript
{
  name: string
  company?: string
  pipeline_stage?: string         // pre-populated from analysis
  detected_persona?: string       // pre-populated from analysis
  source?: "batch_transcript"
  transcript_filename?: string
}
```

When source is "batch_transcript", create a changelog entry:
- action: LEAD_CREATED_FROM_TRANSCRIPT
- detail: investor name + transcript filename + detected persona
  and stage

---

## Step 3 — Backend: Auto-match on batch analysis completion

Update POST /api/recommendation/analyze-batch to run a lead
match automatically for each result where investor_name is
non-null. Run matches in parallel — do not add sequential latency.

Add to each result in the response:
```typescript
{
  filename: string
  investor_name: string | null
  status: "success" | "error"
  analysis?: { ... }
  lead_match?: {
    matched: boolean
    top_match?: {
      lead_id: string
      name: string
      pipeline_stage: string
      detected_persona: string
      confidence: number
    }
    match_count: number
  }
}
```

If investor_name is null, omit lead_match entirely.

---

## Step 4 — Frontend: Batch result card lead management

Update the batch result cards based on lead_match data:

**High-confidence match (confidence ≥ 0.85):**
Green "Likely match" banner:
"Likely match: [Name] — [Stage] · [Persona]"
Buttons: "Confirm match" and "Not this person"

**Low-confidence match (confidence 0.4–0.84):**
Yellow "Possible match" banner listing up to 3 candidates with
confidence scores. User selects one or dismisses.

**No match or investor_name is null:**
Blue "New contact" banner:
"No existing lead found for [Name]"
Button: "Create Lead"

**On "Create Lead" click:**
Open an inline form (not a new page):
- Name: pre-populated from investor_name (editable)
- Company: empty (editable)
- Pipeline stage: pre-populated from detected stage (editable
  dropdown — canonical stages only)
- Persona: pre-populated from detected persona (editable
  dropdown — canonical archetypes only)

On submit: POST /api/leads with source: "batch_transcript"
On success: show green "Lead created — Link" button that
navigates to the recommend flow for the new lead with analysis
pre-populated.

**On "Confirm match" click:**
Record the lead linkage (existing behaviour) and navigate to
the recommend flow pre-populated.

# BUILD 077 — Batch Analysis Depth Parity
Priority: HIGH
Area: Recommendation Engine — Batch Upload

## Overview

The single transcript analyze endpoint returns 7 signal types:
persona, stage, readiness, blocking objections, information gaps,
primary issue, and recommended next action.

The batch analyze endpoint currently returns only 2: persona and
stage. This build brings the batch endpoint to full parity so
agents processing 30,000-contact Aircall transcripts in batches
get the same intelligence as processing a single transcript.

---

## Step 1 — Update the analyze-batch Claude prompt

Update POST /api/recommendation/analyze-batch to use the same
enhanced prompt as POST /api/recommendation/analyze (build 058).

Include in every transcript analysis in the batch:

```
READINESS TO CLOSE
Buying signals include asking about process, next steps, paperwork,
mentioning a specific investment amount, referencing tax year
deadlines, comparing favourably to alternatives, or no unresolved
objections remaining.
Rate as: READY | WARM | NURTURING | COLD
Provide specific evidence from the transcript.

BLOCKING OBJECTIONS
Objections actively preventing progress include needing to speak
to an accountant/IFA/partner, risk concerns expressed more than
once, uncertainty about product mechanics, hesitation language
("not sure", "need to think", "maybe later"), timeline deflection
without a specific date.
For each: name it, quote the evidence, rate severity as
CRITICAL | SIGNIFICANT | MINOR.

INFORMATION GAPS
Topics the investor is confused about or expressing misconceptions
about include questions about EIS/SEIS/VCT mechanics, incorrect
comparisons to ISAs or pensions, confusion about tax relief timing,
misunderstanding of risk or loss relief.
For each: name it, describe the confusion, identify what document
or explanation would address it.

PRIMARY ISSUE
The single most important thing driving next action:
- READY_TO_CLOSE — buying signals present, objections resolved
- OBJECTION_TO_RESOLVE — a specific blocking objection must be
  addressed first
- INFORMATION_GAP — investor needs specific education before
  they can decide
- NEEDS_NURTURING — too early, insufficient signals to act on yet
Provide one sentence: recommended next action for the salesperson.

CALL COMPLETENESS
Assess which of the 4 call framework questions were covered:
Q1: Portfolio composition, Q2: Investment objective,
Q3: EIS/SEIS experience, Q4: What needs to happen to move forward.
Return: questions_covered (integer), missing_signals (array).
```

Also include the Aircall speaker label instructions:
analyse only Investor speaker turns for signals — Agent turns
provide context only and must not be treated as investor signals.

---

## Step 2 — Update the analyze-batch response shape

```typescript
{
  results: Array<{
    filename: string
    investor_name: string | null
    status: "success" | "error"
    lead_match?: { ... }
    analysis?: {
      // existing fields unchanged
      persona: string
      persona_confidence: number
      stage: string
      stage_confidence: number
      objections: string[]
      evidence: string[]

      // new fields — parity with single endpoint
      readiness: "READY" | "WARM" | "NURTURING" | "COLD"
      readiness_evidence: string[]
      blocking_objections: Array<{
        name: string
        evidence: string
        severity: "CRITICAL" | "SIGNIFICANT" | "MINOR"
      }>
      information_gaps: Array<{
        topic: string
        description: string
        suggested_document_type: string
      }>
      primary_issue: "READY_TO_CLOSE" | "OBJECTION_TO_RESOLVE" |
                     "INFORMATION_GAP" | "NEEDS_NURTURING"
      recommended_next_action: string
      call_completeness: {
        questions_covered: number
        questions_total: 4
        missing_signals: string[]
        confidence_impact: string
      }
    }
    error?: string
  }>
}
```
