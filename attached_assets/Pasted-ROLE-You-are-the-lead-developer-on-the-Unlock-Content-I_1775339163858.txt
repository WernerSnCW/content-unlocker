ROLE: You are the lead developer on the Unlock Content Intelligence Platform. You have full context of the existing codebase.

CONTEXT: The platform currently approves content at the document level. A new architectural layer is required: Approved Content Units (ACUs). ACUs approve facts, framings, references, explanations, and qualifiers at source — before generation. Generated documents then draw from locked units rather than producing free-form text. This makes compliance architectural rather than instructional.

The existing molecule store concept is superseded by this framework. ACUs are the full implementation of that instinct.

TASK: Build the ACU framework in four parts, in this order. Complete each part before moving to the next. Confirm completion of each part before proceeding.

---

PART 1 — DATABASE SCHEMA

Create a new table: approved_content_units

Fields:
- id TEXT PRIMARY KEY — format: acu_{descriptor} e.g. acu_eis_loss_relief
- type TEXT NOT NULL — one of: fact / framing / reference / explanation / qualifier / prohibited
- content TEXT NOT NULL — the locked text
- status TEXT NOT NULL — one of: DRAFT / APPROVED / LOCKED / LEGAL_PENDING / NAMING_PENDING / SUPERSEDED
- source TEXT — citation (required before APPROVED status can be set)
- approved_by TEXT — tom_king / legal / compliance
- approved_date TEXT
- version INTEGER DEFAULT 1
- expression_variants JSONB — array of {audience, stage, text}
- documents_referencing JSONB — array of document IDs
- cascade_on_change BOOLEAN DEFAULT true
- notes TEXT

Status rules (enforce in code, not just convention):
- Cannot move to APPROVED without source field populated
- Cannot move to LOCKED without being APPROVED first
- LOCKED content cannot be edited — only a new version can be created
- LEGAL_PENDING and NAMING_PENDING are non-injectable — generation engine must not use these
- PROHIBITED type units are always non-injectable and trigger QC fail on detection

PART 2 — API ENDPOINTS

Add to the existing Express API:

GET  /api/acu                    — list all units, filterable by status and type
GET  /api/acu/:id                — single unit with full version history
POST /api/acu                    — create new unit (status: DRAFT)
PATCH /api/acu/:id/approve       — move DRAFT → APPROVED (requires source to be populated)
PATCH /api/acu/:id/lock          — move APPROVED → LOCKED (irreversible)
POST /api/acu/:id/version        — create new DRAFT version (old LOCKED version remains active)
GET  /api/acu/prohibited         — list all PROHIBITED units (used by QC engine)
GET  /api/acu/injectable         — list all LOCKED units (used by generation engine)
PATCH /api/acu/:id/cascade       — trigger document review flag for all documents_referencing this unit

PART 4 — INTEGRATION

4a. QC ENGINE UPDATE
Add two new check types to the QC engine:
- ACU_PROHIBITED_PRESENT — scan document content against all PROHIBITED ACUs. If any prohibited content is detected, fail immediately. This check runs before all other checks.
- ACU_LOCKED_ABSENT — for each LOCKED ACU relevant to the document type, verify the locked content is present. If a required ACU is paraphrased rather than verbatim, flag as a warning (not a hard fail — some flexibility in connective tissue is acceptable, but the core fact must be present).

4b. GENERATION PIPELINE UPDATE
When generating a brief, the tool must:
1. Identify which ACUs are relevant based on document type, belief, and persona
2. Inject all relevant LOCKED ACUs as fixed content blocks in the generation prompt
3. Inject all relevant PROHIBITED ACUs as explicit prohibitions in the generation prompt
4. Claude generates the connective tissue around the fixed content — it does not rewrite the locked units

Format for injection in generation prompt:
```
LOCKED CONTENT — inject verbatim, do not paraphrase:
[acu_eis_loss_relief content]
[acu_capital_at_risk content]

PROHIBITED CONTENT — must not appear:
[acu_22p_prohibited content]
[acu_78x_prohibited content]
```

4c. UI — ACU MANAGEMENT PAGE
Add a new page: /acu

Display:
- Summary counts: Locked / Approved / Draft / Legal Pending / Naming Pending / Prohibited
- Filterable list by status and type
- Each row: id / type / status pill / source / approved_by / documents_referencing count
- Click to expand: full content, version history, expression variants, cascade button
- Actions per status:
  - DRAFT: Approve button (disabled if source empty) / Edit / Delete
  - APPROVED: Lock button (with confirmation modal showing blast radius) / Edit
  - LOCKED: New Version button / View History
  - LEGAL_PENDING: View only — no actions until legal confirms
  - NAMING_PENDING: View only — Mark as Confirmed button (requires written confirmation)

Confirmation modal for Lock action:
"You are about to lock this content unit. This action cannot be undone. Once locked, this exact text will be injected verbatim into all generated documents that reference it. [N] documents currently reference this unit. Locking a new version will flag them for review. Confirm?"
