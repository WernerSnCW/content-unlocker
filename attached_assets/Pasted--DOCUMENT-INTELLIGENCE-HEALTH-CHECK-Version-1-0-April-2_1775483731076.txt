# DOCUMENT INTELLIGENCE HEALTH CHECK
Version: 1.0 | April 2026

Run this prompt in Replit AI against the Unlock Intel codebase.
It audits every document in the registry across all dimensions
that matter for content delivery to investors.

---

## WHAT THIS CHECKS

For every CURRENT document in the system:

1. **Identity** — does the document know what it is?
   Tier, category, name, file_code, lifecycle_status — all populated correctly?

2. **Targeting** — does the document know who it's for?
   persona_relevance, pipeline_stage_relevance — populated and correct?

3. **Belief alignment** — does the document know what it builds?
   belief_targets — mapped to the right beliefs with correct state transitions?

4. **Compliance** — is the document clean?
   review_state, quality_score, any prohibited values in content?

5. **Propagation integrity** — does the document sit correctly in the cascade?
   upstream_dependencies, tier hierarchy — connected correctly?

6. **Pipeline stage fit** — is it being recommended at the right moment?
   pipeline_stage_relevance vs where leads actually receive it (send_log data)?

7. **Content completeness** — does it have content to deliver?
   content field populated, not empty, not a stub?

8. **Delivery record** — has it ever been sent? To whom?
   send_log analysis across all leads — is it being used?

---

## HOW TO RUN

Work through each section in order. For each section:
- Run the queries specified
- Inspect results
- Report findings using the output template
- Flag anomalies specifically — name the document and the issue

Do not summarise generically. Every finding must name a specific document.

### 4. COMPLIANCE STATUS

Check the compliance health of every CURRENT document.

```sql
SELECT
  d.id,
  d.name,
  d.tier,
  d.review_state,
  d.quality_score,
  d.quality_score_detail,
  COUNT(t.id) FILTER (WHERE t.status = 'Open' AND t.type = 'Review') as open_review_tasks,
```

```sql
  name,
  tier,

    WHEN tier = 1 AND (upstream_dependencies IS NULL OR upstream_dependencies = '[]'::jsonb)
```

### 6. PIPELINE STAGE FIT

Are documents being recommended and sent at the right stage?
Compare declared pipeline_stage_relevance against actual send data.

```sql
-- Analyse send_log across all leads
-- For each document that has been sent, identify the pipeline_stage
-- of the lead at the time of send (from stage_history if available,
-- or current pipeline_stage as a proxy)
-- Compare against the document's declared pipeline_stage_relevance
-- Flag documents sent outside their declared stage relevance
SELECT
  d.id,
  d.name,
  d.tier,
  d.pipeline_stage_relevance,
  COUNT(*) as times_sent
FROM documents d
JOIN leads l ON l.send_log::text LIKE '%' || d.id || '%'
WHERE d.lifecycle_status = 'CURRENT'
GROUP BY d.id, d.name, d.tier, d.pipeline_stage_relevance
ORDER BY times_sent DESC;
```

|---|---|---|---|
| [name] | [stages] | [actual stage] | [lead_id] |

### 7. CONTENT COMPLETENESS

Is there actual content to deliver?

```sql
SELECT
  id,
  name,
  tier,
  lifecycle_status,
  CASE
    WHEN content IS NULL THEN 'NULL CONTENT'
    WHEN LENGTH(content) < 100 THEN 'STUB — under 100 chars'
    WHEN LENGTH(content) < 500 THEN 'THIN — under 500 chars'
    ELSE 'OK — ' || LENGTH(content) || ' chars'
  END as content_status,
  LENGTH(content) as content_length
FROM documents
WHERE lifecycle_status = 'CURRENT'
ORDER BY content_length ASC NULLS FIRST;
```
