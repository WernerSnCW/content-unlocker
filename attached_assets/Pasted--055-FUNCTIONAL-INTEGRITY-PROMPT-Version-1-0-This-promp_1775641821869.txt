# 055 — FUNCTIONAL INTEGRITY PROMPT
Version: 1.0

---

This prompt assesses whether the significant functions in this codebase
are achieving their stated purpose — not just whether they run without
errors, but whether they produce the right output, use the right inputs,
and connect to the right parts of the system.

050 (Health Check) tells you if a feature is WORKING or BROKEN.
055 tells you if a feature is ACHIEVING ITS GOAL or HOLLOW.

A function is HOLLOW when it:
- Responds without errors but ignores available context it should use
- Generates output that could have been produced without its inputs
- Exists in isolation when it should be connected to other system state
- Produces the same result regardless of what data it receives

Run this after 050 confirms the system is in a working state.
Do not run 055 to diagnose errors — use 050 for that.

---

## HOW TO RUN THIS ASSESSMENT

For each significant function or workflow listed below:

1. Read the code
2. Identify what inputs are available vs what inputs are actually used
3. Identify what outputs are produced vs what outputs should be produced
4. Identify what system state the function could use but doesn't
5. Run a live test with specific inputs and inspect the output
6. Assess: is this function achieving its goal, or is it hollow?

## WHAT TO ASSESS

Run this assessment against every function, endpoint, or workflow that:
- Takes a lead_id or document_id as input and generates output
- Produces content, recommendations, or analysis
- Is supposed to personalise its output based on stored data
- Connects two or more parts of the system

Do NOT assess:
- Pure CRUD endpoints (GET/POST/PATCH that simply read/write data)
- Schema definitions, migrations, seed functions
- Utility functions with no business logic (fuzzyMatch, format, etc.)
- Authentication or infrastructure code

When uncertain whether to assess a function: assess it.

---

## EVIDENCE STANDARDS

For ACHIEVING GOAL verdict:
- Live test showing output changes meaningfully when input context changes
- Code inspection confirming all relevant fields are read and used

For HOLLOW verdict:
- Show that a function with a lead_id produces the same output as without it
- OR show that available context (intelligence profile, belief states, etc.)
  is not fetched despite being relevant to the function's goal
- Code inspection alone is sufficient for HOLLOW if the data fetch is
  clearly absent from the code

For PARTIAL verdict:
- List exactly which inputs are used and which are ignored
- Show which output fields are present and which are missing

For UNKNOWN verdict:
- State what specific test would be needed to determine the verdict
- State what data would need to exist to run that test

---

## SCORING NOTE

The goal of this assessment is to find HOLLOW functions — not to score
the codebase. A system with 3 HOLLOW functions and 15 ACHIEVING GOAL
functions is doing well. A system where every function is PARTIAL is
systematically underperforming even if nothing is broken.

The most dangerous finding is a function that produces plausible-looking
output that is actually generic — the user cannot tell the difference, but
the output is not informed by what the system knows.

---

## SCHEDULING

Run 055 (Functional Integrity):
- After every major feature build that adds new data to the system
  (belief states, intelligence profiles, new tables) — check whether
  existing functions have been updated to use the new data
- When a feature is reported as working but producing generic or
  unhelpful output
- Before a product review (070) — 055 identifies the functional gaps
  that 070 will surface as UX problems

055 is a Replit AI prompt. Outputs are advisory. Any fix identified must
be explicitly pulled into a future Step 2 idea before being built.
