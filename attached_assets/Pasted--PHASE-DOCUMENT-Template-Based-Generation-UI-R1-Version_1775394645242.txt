## Section 1 — Feature

Adds a “From Template” mode to the existing Content Generation page (`artifacts/unlock-intel/src/pages/generate.tsx`) via a two-tab toggle. When the user selects “From Template”, they choose from the 22 registered output templates, see the template’s section structure and compliance constraints (required/prohibited ACUs with content previews), provide a document name and optional context brief, and submit to the existing template-aware generation endpoint. The existing “Custom” tab and all its logic remain completely unchanged.

-----

## Section 2 — Why this is necessary

Every document generated through the current Content Generation page bypasses the Template Registry entirely. `POST /generation/generate` (the current endpoint) applies no template section definitions, no required ACU enforcement, no prohibited ACU checking, and no parent template inheritance. The backend already has a complete compliant generation path — `generateFromTemplate()` in `artifacts/api-server/src/lib/generationEngine.ts`, served by `POST /generation/from-template` — but it has zero frontend exposure. As a result, the compliance value of the Template Registry (22 templates, required ACU injection at 95% match rate, prohibited ACU absence enforcement, section-level QC) is entirely invisible to users and produces no governed output. This is an output quality problem, not a UX preference.

-----

## Section 3 — Current state

- `artifacts/unlock-intel/src/pages/generate.tsx`: single-mode page with a free-text form (document name, type, requirements, personas) submitting to `POST /generation/generate`. Contains existing QC result display (pass/fail badge, issues list, regenerate button, promote button). No template awareness. No tab structure.
- `artifacts/api-server/src/routes/generation/index.ts`: implements `POST /generation/from-template`, which calls `generateFromTemplate()`. Exact request body shape must be confirmed from this file — it is not documented in the scan.
- `artifacts/api-server/src/lib/generationEngine.ts`: `generateFromTemplate(request)` accepts `template_id`, optional `context`, and optional `channel_temperature`. Merges parent template ACUs, injects locked ACU content verbatim, runs `checkTemplateCompliance()`. Response shape must be confirmed against the Custom mode response shape before QC display reuse is confirmed.
- `artifacts/api-server/src/routes/templates/index.ts`: `GET /templates` returns all 22 templates. `GET /templates/:id` returns a single template with merged parent sections.
- `artifacts/api-server/src/routes/acu/index.ts`: `GET /acu` returns all ACUs with full content, status, and type.
- `lib/api-client-react/src/generated/`: auto-generated hooks — whether hooks for `GET /templates` and `POST /generation/from-template` exist is unconfirmed. Manual fetch is the fallback.

## Section 6 — Acceptance criteria

AC1. The Content Generation page (`/generate`) renders a two-tab toggle with tabs labelled “From Template” and “Custom”. The “From Template” tab is active by default on page load.

AC2. The “Custom” tab renders the existing generation form with identical fields, layout, and submit behaviour as before this build. Submitting from the Custom tab calls `POST /generation/generate` with the same request body fields as prior to this build (document name, type, requirements, personas).

AC3. Switching from “From Template” to “Custom” and back clears all From Template form state (selected template, document name, context, and any prior generation result from the From Template flow). Custom mode state is unaffected by tab switches.

AC4. On “From Template” tab activation, `GET /templates` is called exactly once. The template picker displays all returned templates grouped by the `output_type` field.

AC5. Selecting a template from the picker calls `GET /templates/:id` for the selected template’s ID. The detail panel renders: (a) section names from the `sections` array, (b) required ACU content previews truncated to 100 characters with “…” if the content exceeds 100 characters, (c) prohibited ACU content previews truncated to 100 characters with “…”. Manual test required: live app with seeded templates and ACUs.

AC6. For any required or prohibited ACU ID that cannot be matched in the `GET /acu` response, the detail panel displays the raw ACU ID in greyed text with the label “Content unavailable”. No unresolved IDs are silently omitted.

AC7. `GET /acu` is called once per page session and cached in component state. It is not re-fetched on subsequent template selections within the same session.

AC8. Submitting the From Template form with an empty document name field is blocked. An inline validation message “Document name is required.” is displayed adjacent to the document name field.

AC9. Submitting the From Template form with a valid document name and a selected template calls `POST /generation/from-template` with the confirmed request body shape (template_id, document name, and context if provided). Manual test required: live app; inspect network request payload.

AC10. On successful generation via the From Template flow, the QC result display renders below the form showing pass/fail status, issues list (if any), regenerate button, and promote button — identical in appearance and behaviour to the Custom mode QC display. Manual test required: live app with seeded templates and ACUs.

AC11. If `GET /templates` returns an error, the template picker area displays: “Could not load templates. Please try again.” If `GET /templates/:id` returns an error, the detail panel displays: “Could not load template details.” If `POST /generation/from-template` returns an error, an inline error is displayed below the submit button.

AC12. No change to `artifacts/api-server/src/routes/generation/index.ts`, `artifacts/api-server/src/lib/generationEngine.ts`, or any other backend file. Manual test required: confirm git diff contains no changes to files outside `artifacts/unlock-intel/src/pages/generate.tsx` and any new component files created in `artifacts/unlock-intel/src/`.

## Section 9 — Assumptions

- `GET /templates` response includes an `output_type` field on each template object, consistent with `outputTemplatesTable` schema (`output_type text`). If the field is named differently in the actual response, the grouping logic must use the correct field name — inspect the response and adjust.
- `GET /acu` returns a flat array of all ACUs including `id`, `content`, and `type` fields without pagination. If the response is paginated, a full fetch strategy must be confirmed before proceeding.
- The `sections` JSONB field in the template response contains an array of objects each with a `name` or `title` field. If the section objects use a different field name, use whichever field is present.
- `POST /generation/from-template` returns a response shape compatible with `POST /generation/generate` — this is the highest-risk assumption and is gated at 5.1.
- The existing QC display in `pages/generate.tsx` is self-contained enough to be rendered conditionally from either mode without refactoring — confirmed by the scan showing the page handles mutations via React Query `useMutation` states.
