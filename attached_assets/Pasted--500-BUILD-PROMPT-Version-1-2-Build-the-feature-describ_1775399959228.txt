## Section 1 — Feature

Fixes two visible product gaps in the Unlock Intel frontend. R2 adds a lead
creation modal to the Lead Management page, triggered by the existing "New Lead"
button that currently produces a 404. R3 restructures the sidebar navigation from
a flat 13-item list into four labelled groups (Operations, Content, Governance,
System). Both changes are frontend-only with no backend modifications and no new
API endpoints.

---

## Section 2 — Why this is necessary

R2: The "New Lead" button appears in two places — the Lead Management page
(/leads) and the Dashboard Quick Actions panel. Both link to /leads/new which
matches no route and produces a 404. There is no way to create a lead from the UI
despite POST /leads being fully implemented on the backend.

R3: The sidebar has 13 items in a flat list with no grouping or hierarchy.
Related features are scattered — "Content Generation" and "Content Gaps" are
separated by "Changelog"; "Call Prep" and "Recommendation Engine" are sequential
workflow steps positioned as independent items. The 070 product review rated this
MEDIUM severity friction affecting all users on every page load.

---

## Section 3 — Current state

  with a "New Lead" button. Button currently links to /leads/new which has no
  route handler. Uses useListLeads({ search }) hook.
- artifacts/unlock-intel/src/pages/dashboard.tsx: Quick Actions panel has an

  layout component, or a dedicated sidebar file. Build agent must locate before

## Section 6 — Acceptance criteria

AC1. On the Lead Management page (/leads), clicking "New Lead" opens a modal
containing inputs for name, company, pipeline stage (Select), and source.
The modal does not navigate to /leads/new.

AC2. Submitting the lead creation modal with an empty name field is blocked.
An inline error "Name is required." is displayed. The form is not submitted.

AC3. Submitting the modal with a valid name calls POST /leads with the confirmed
request body. On success the modal closes, form fields are cleared, and the lead
appears in the leads table without a page reload. Manual test required: live app;
verify lead appears in table and in the database.

AC4. Submitting the modal with a valid name and all optional fields populated
calls POST /leads with all provided fields included in the request body.
Manual test required: live app; inspect network request payload.

AC5. If POST /leads returns an error, the modal remains open and displays
"Could not create lead. Please try again." below the submit button.

AC6. The "Add New Lead" Quick Action on the Dashboard navigates to /leads
(not /leads/new). Clicking it does not produce a 404.

AC7. The sidebar navigation renders four labelled groups: "Operations",
"Content", "Governance", "System". Each group label is visually distinct
from nav items (smaller, muted, non-clickable). Manual test required: visual
inspection in browser.

AC8. All 13 existing sidebar navigation items are present after the restructure.
No items have been removed. All route paths are unchanged. Manual test required:
click each item and verify it navigates to the correct route.

AC9. The active route highlight applies correctly to all nav items after
restructure — the currently active page item is highlighted regardless of
which group it is in. Manual test required: navigate to each page and verify
correct item is highlighted.

AC10. No changes to any file in artifacts/api-server/ or lib/. Manual test
required: confirm git diff contains no backend file changes.
