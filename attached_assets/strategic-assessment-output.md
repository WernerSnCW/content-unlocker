## PRODUCT GOALS (operator-supplied)

**What this tool is for:**
Unlock is an internal intelligence platform for a UK fintech company that manages investor relations at scale. It automates the process of analysing investor call transcripts, recommending the right compliance-approved documents to send, and generating new content using AI — all while enforcing strict regulatory compliance through Approved Content Units (ACUs) and a 19-check quality control system. It also tracks investor belief states, manages multi-channel campaigns, and detects content gaps across an 84-document library.

**Who uses it:**
The primary user is an internal investor relations / content operations manager at the company. They are non-technical but highly domain-knowledgeable — they understand investor personas, pipeline stages, EIS/VCT tax relief products, and compliance requirements. They use the tool daily to decide what to send investors, generate compliant content, and maintain the document library.

**What success looks like:**
1. The recommendation engine accurately identifies investor persona, pipeline stage, and objections from call transcripts, then ranks the right documents to send.
2. Generated content always includes Locked ACUs and never includes Prohibited ACUs — compliance is non-negotiable.
3. The document library stays healthy: propagation cascades flag stale content, the work queue surfaces issues for resolution, and content gaps are identified before they cause problems.
4. The platform reduces manual effort — bulk imports, auto-generated email drafts, campaign asset generation, and PDF/Google Docs exports all work reliably.
5. Data integrity is maintained across leads, beliefs, documents, and campaigns with full audit logging.

**What it is not for:**
- Not a customer-facing tool — investors never see or interact with it.
- Not a general-purpose CRM — lead management exists only to support content targeting and recommendation.
- Not a document editor — content is generated by AI and reviewed, not manually authored within the tool.
- Not a marketing automation platform — campaigns are planned and generated here but actual delivery happens externally.

| Area | Appropriate | Notes |

**Overall architecture verdict:** WELL-SUITED

The AI integration pattern (structured prompt → validate output → write to DB → changelog) is consistent and robust. The two areas that are genuinely under-built are file storage (filesystem instead of object storage) and async processing (no recovery for interrupted sessions). Neither blocks current functionality but both would need fixing before production deployment.

### 4. Enhancements

#### `Surface buried intelligence data in UI`
**Addresses:** G1 (recommendation accuracy depends on operators having context), G4 (reduces manual effort by showing pre-computed data)
**What to build:** Add to lead-detail page: SPIN analysis section (4 fields), qualification boolean checklist (9 fields), belief evidence/source/confidence in belief detail panel, belief transitions timeline.
**Effort estimate:** MEDIUM
**Priority:** HIGH — This data is already generated and stored. Operators are making calls without seeing the SPIN analysis, qualification flags, or belief evidence that the system has already computed. This is the highest-value improvement because it requires zero backend work.

#### `Fix inv_opp_feb26 compliance gap and prevent recurrence`
**Addresses:** G2 (compliance is non-negotiable)
**What to build:** Immediately flag inv_opp_feb26 for review. Add a post-auto-fix verification step that re-scans the document for ALL prohibited values after applying fixes (currently, auto-fix applies a targeted replacement but doesn't verify the rest of the document is clean).
**Effort estimate:** LOW
**Priority:** HIGH — A document with prohibited pricing is marked CLEAN with no review task.

#### `Pass belief state map into document ranking prompt`
**Addresses:** G1 (recommendation accuracy)
**What to build:** Include the lead's current belief state map in the Claude ranking prompt so the AI can factor belief progression into its ranking decision, not just as post-hoc enrichment. Documents that advance a belief from PARTIAL to ESTABLISHED should rank higher than equivalent documents that don't advance any belief.
**Effort estimate:** LOW
**Priority:** MEDIUM — The belief enrichment is already post-hoc applied. Passing it to Claude would improve ranking quality but the current approach is functional.

#### `Add work queue session status distinction`
**Addresses:** G3 (library health), G5 (data integrity)
**What to build:** Distinguish between "analysis complete" (findings generated, ready for operator review) and "fully resolved" (all findings accepted/skipped). Currently both map to COMPLETE, leaving 58 orphaned PENDING findings in 2 sessions.
**Effort estimate:** LOW
**Priority:** MEDIUM

#### `Close stale task/document mismatches`
**Addresses:** G5 (data integrity)
**What to build:** Add a reconciliation step after work queue card acceptance: if a document is re-flagged by propagation after a task is completed, the old Done task should not appear as a false signal. Either auto-close the Done task when re-propagation creates a new Open task, or add a "superseded by propagation" status.
**Effort estimate:** LOW
**Priority:** MEDIUM

#### `Populate belief evidence trails`
**Addresses:** G5 (data integrity — audit compliance)
**What to build:** When the belief analysis endpoint (POST /leads/:leadId/beliefs/analyze) establishes a belief, require it to write evidence (the transcript excerpt), evidence_source (transcript analysis), and confidence score. Currently 8 beliefs are ESTABLISHED with no evidence or confidence — these drive gating decisions without audit trail.
**Effort estimate:** LOW
**Priority:** MEDIUM

#### `Migrate PDF storage to object storage`
**Addresses:** G4 (export reliability)
**What to build:** Replace writeFile/readFile PDF storage with Replit Object Storage (or equivalent). Generated PDFs and exported documents currently use local filesystem and will be lost on deployment restart.
**Effort estimate:** MEDIUM
**Priority:** MEDIUM — Not blocking development but blocks reliable production deployment.

### 6. Out-of-scope flags

- **Email delivery integration:** The platform generates email drafts but does not send them. Adding actual email delivery (SMTP, SendGrid, etc.) would expand scope beyond "campaigns are planned and generated here but actual delivery happens externally." If delivery integration is desired, it should be a separate architectural decision.
- **Investor-facing portal:** Any feature that exposes the platform or its content directly to investors would contradict the "strictly internal" constraint.
- **Real-time Pipedrive/Aircall sync:** The platform has a Pipedrive/Aircall setup guide (document 210) but no live CRM integration. Adding real-time sync would expand the tool into CRM territory. If desired, it should be scoped as an integration project, not a feature addition.
- **lib/ directory refactoring to workspace packages:** Moving lib/personas.ts, lib/call-questions.ts, and lib/coverage-matrix.ts into proper @workspace packages would improve maintainability but is an architectural change that should be reviewed separately.

### 7. Deprecation candidates (optional)

- **`GET /documents/:id/download-pdf`:** Appears to be an older PDF download endpoint. `GET /documents/:id/export-pdf` and `POST /documents/:id/generate-pdf` serve the same purpose with more features. The download-pdf endpoint should be verified for usage and removed if redundant.
- **`notes_legacy` field on leads:** Field exists in schema but has no UI representation and no API write path. If notes have been migrated to the `notes` field, this legacy field should be dropped.
