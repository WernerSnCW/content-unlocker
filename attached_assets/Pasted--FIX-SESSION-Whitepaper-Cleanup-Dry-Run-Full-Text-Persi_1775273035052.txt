## Fix 2 — Feature update dry_run implementation

### The problem
The feature update endpoint accepts a dry_run parameter but
ignores it entirely. Every call writes changelog entries and
flags documents as REQUIRES_REVIEW, even when the caller is
just exploring what would be affected. This caused 6 core
documents to be incorrectly flagged during the health check.

### The fix

In the POST /api/content/feature-update route handler:

1. Read the dry_run boolean from the request body
   (default false if not provided)

2. If dry_run is true:
   - Run all detection logic (tier propagation, semantic match,
     keyword match, compliance match) exactly as normal
   - Collect the list of affected documents and detection reasons
   - DO NOT write any changelog entries
   - DO NOT update any document review_state
   - Return the same response shape as a real run, but add:
     `"dry_run": true` to the response root
     `"message": "Dry run only — no documents were modified"`

3. If dry_run is false (or not provided):
   - Existing behaviour unchanged — write changelog, flag docs

The response shape must be identical whether dry_run is true
or false, so callers can use the same parsing logic.

### Update the OpenAPI spec

Add dry_run to the FeatureUpdateBody schema:
```yaml
dry_run:
  type: boolean
  default: false
  description: >
    If true, run detection only — do not write any changelog
    entries or modify document review states. Returns the same
    response shape with dry_run: true in the response.
```

Add dry_run and message to the FeatureUpdateResponse schema.

Regenerate API clients after spec update.
