## Fix 2 — Seed function sets content: null for all documents
**Priority:** HIGH — blocks Google Docs exports and content previews
**File:** dataManager.ts — the seedDatabase function

### The problem
The seed function explicitly sets content: null for all documents.
Content markdown files exist on disk at:
  artifacts/api-server/src/data/content/

These files are not read during seeding. As a result:
- All Google Docs exports send empty content
- Content Preview on all document detail pages shows "No content available"
- The Google Docs round-trip mechanism works but operates on empty documents

### The fix
Update the seed function to load content from the content directory
into the database when seeding documents.

Before implementing, inspect the content directory and identify the
naming convention used by the content files. Do not assume — read the
directory first and establish whether files are named by document ID,
document title slug, or some other pattern.

The loading logic must:
- For each document being seeded, look for a corresponding content file
  using the naming convention identified above
- If a matching file is found, read its contents and set the content
  field to that string
- If no matching file is found, set content to null — do not throw,
  handle missing files gracefully
- Log which documents loaded content and which fell back to null so
  the result is visible

Do not change the document registry schema or any endpoint.
Only the seed function should change.

### Confirm with live tests

Test 1 — Content loads on seed:
- Wipe and reseed the database
- Call GET /api/documents/:id for at least 5 documents
- Confirm the content field is non-null for documents that have
  matching content files
- Confirm documents without matching files have content: null and
  did not cause the seed to fail

Test 2 — Google Docs export sends real content:
- Call POST /api/gdocs/export/:id for a document that now has content
- Confirm the created Google Doc contains the document's actual content,
  not an empty string
- Confirm GET /api/gdocs/status/:id returns linked: true and accessible: true

Test 3 — Content preview renders:
- Open the document detail page for a seeded document with content
- Confirm the Content Preview section renders the content text
- Confirm it does not show "No content available"

## Fix 3 — Lead PATCH crashes with 500 on unrecognised fields
**Priority:** MEDIUM — causes crashes instead of clean errors
**File:** leads/index.ts — the PATCH /api/leads/:id handler

### The problem
The PATCH handler builds an updates object from the request body.
It only processes four fields: pipeline_stage, detected_persona,
archived, and company. If the request body contains only fields
outside this set (e.g. notes, or any unknown field), the updates
object is empty and Drizzle throws "Error: No values to set" — a
500 crash. The notes field in particular is likely to be sent by
the frontend but is silently unhandled.

### The fix — two changes required

**A. Add notes field handling**
Add notes to the list of fields processed by the PATCH handler.
Notes should be written to the lead record.

Also add updated_at to every PATCH so the lead's last-modified
timestamp is always refreshed on any successful update.

**B. Guard against empty updates**
After building the updates object, check if it is empty before
passing it to Drizzle. If empty, return a 400 response with a
clear error message:
  { "error": "No valid fields provided for update" }

Do not let an empty updates object reach the database layer under
any circumstances.

Do not change the Zod input schema beyond what is needed to accept
the notes field. Do not change any other endpoint.

### Confirm with live tests
