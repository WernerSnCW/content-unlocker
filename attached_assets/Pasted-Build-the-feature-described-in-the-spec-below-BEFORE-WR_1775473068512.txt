## Section 1 — Feature

Extends `POST /recommendation/rank` with three belief-aware behaviours when a `lead_id` is provided: (1) hard gate exclusions — documents gated behind F0 are removed before Claude ranking; (2) intelligence-derived matrix flags — `eis_familiar` and `adviser_mentioned` auto-populated from `lead_intelligence`; (3) belief context on ranked output — each ranked document gains `belief_targeted`, `current_state`, `state_after_send`, `gate_it_unlocks`. When no `lead_id` provided, behaviour is completely unchanged.

---

## Section 2 — Why this is necessary

Without this phase, the recommendation engine ranks documents without awareness of belief states or gates. A lead without F0 could receive Pack 1. The intelligence table stores `already_done_eis` and `ifa_involved` but rank ignores them.

---

## Section 3 — Current state

From the Phase 010c Step 1 scan of `POST /recommendation/rank`:

- Input: `lead_id` (optional), `detected_persona`, `pipeline_stage`, `transcript_summary`, `objections`, `eis_familiar` (bool), `iht_confirmed` (bool), `adviser_mentioned` (bool)
- When `lead_id` provided: fetches lead from `leadsTable` for send history only — does NOT fetch `lead_intelligence` or `lead_beliefs`
- Filter pipeline: lifecycle → review state → stage → persona → already-sent → usage matrix exclusion
- No belief-based gate logic exists
- `eis_familiar`/`iht_confirmed`/`adviser_mentioned` are request-supplied only
- Ranked document output: `document_id`, `file_code`, `name`, `description`, `tier`, `priority`, `rank`, `relevance_score`, `worth_it`, `rationale` — no belief fields
- `documentsTable` has `belief_targets` column (added in 010a, defaulting to `[]`) — not used in rank
