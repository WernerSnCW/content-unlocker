## Section 1 — Feature

When a document propagation event flags downstream documents as REQUIRES_REVIEW, a new helper function automatically creates "Review: [document name]" tasks for each flagged document, skipping any document that already has an Open or In Progress review task. This connects the existing propagation workflow to the task manager built in Phase A, so document review work surfaces automatically in the Task Board without manual entry.

---

## Section 2 — Why this is necessary

Phase A built the task manager infrastructure but tasks must currently be created manually. The primary use case for the task manager is document review — specifically documents flagged REQUIRES_REVIEW by the propagation engine. Without Phase B, operators must manually create review tasks after every propagation event. This build closes the gap.

## Section 5 — Required behaviour

### 5.1 — New helper file

File: `artifacts/api-server/src/lib/taskHelpers.ts` (create new file)

Define and export one function:

```typescript
export async function createReviewTasksForPropagation(targets: PropagationTarget[]): Promise<void>
```

Import requirements:
- `PropagationTarget` type from `'./propagation'` — if `PropagationTarget` is not currently exported from `propagation.ts`, add the `export` keyword to the interface definition in `propagation.ts`. This is the only permitted change to `propagation.ts` in this build — no logic changes.
- `db` from `@workspace/db`
- `tasksTable`, `changelogTable` from `@workspace/db`
- `eq`, `and`, `ne` from `drizzle-orm`
- `randomUUID` from `crypto`

**Function behaviour — for each target in `targets` (iterate in order):**

Step 1: Query `tasksTable` for any existing task matching ALL of:
- `linked_document_id = target.document_id`
- `type = 'Review'`
- `status != 'Done'`

Step 2: If a matching task exists — skip this target entirely. No insert, no changelog.

Step 3: If no matching task exists — insert a new task:
```
id: randomUUID()
title: 'Review: ' + target.title
status: 'Open'
type: 'Review'
linked_document_id: target.document_id
```

Step 4: Write changelog entry inline immediately after the insert:
```typescript
await db.insert(changelogTable).values({
  id: randomUUID(),
  action: 'TASK_CREATED',
  document_id: target.document_id,
  details: 'Review task auto-created: ' + target.title,
  triggered_by: 'propagation'
})
```

Step 5: Error handling — wrap each target's steps 1–4 in try/catch:
```typescript
try {
  // steps 1–4 for this target
} catch (err) {
  console.error('taskHelpers: failed to create review task for document', target.document_id, err);
  // continue to next target
}
```

The function returns `Promise<void>`. Errors for individual targets are isolated — one failure does not abort processing of remaining targets.

## Section 6 — Acceptance criteria

AC1. After calling `POST /documents/:id/propagate` on a document that has at least one downstream dependent with no existing Open or In Progress review task, the tasks table contains one new task per such dependent with: `title = 'Review: [document name]'`, `type = 'Review'`, `status = 'Open'`, `linked_document_id = [dependent document id]`. Manual test required: use a document known to have downstream dependents, call `POST /documents/:id/propagate`, then call `GET /api/tasks` and confirm new Review tasks appear.

AC2. After calling `POST /documents/:id/propagate` on a document where one downstream dependent already has an Open review task linked to it, no duplicate task is created for that dependent. Manual test required: create a Review task with `status: 'Open'` and `linked_document_id` set to a downstream document's id, trigger propagation, confirm the task count for that document remains 1.

AC3. After calling `POST /documents/:id/propagate` on a document where one downstream dependent has an In Progress review task linked to it, no duplicate task is created for that dependent. Manual test required: create a Review task with `status: 'In Progress'` and `linked_document_id` set to a downstream document's id, trigger propagation, confirm no new task is created.

AC4. After calling `POST /documents/:id/propagate` on a document where one downstream dependent has a Done review task linked to it, a new Open review task IS created for that dependent. Manual test required: create a Review task with `status: 'Done'` and `linked_document_id` set to a downstream document's id, trigger propagation, confirm a new Open Review task is created.

AC5. The `POST /documents/:id/propagate` response shape is unchanged: returns 200 with `{ flagged_document_ids: string[], changelog_entry_ids: string[] }`. Manual test required: call `POST /documents/:id/propagate` and confirm the response contains exactly these two fields.

AC6. For each auto-created task, a row exists in the changelog table with `action = 'TASK_CREATED'`, `document_id` equal to the linked document's id, `details` starting with `'Review task auto-created: '`, and `triggered_by = 'propagation'`.

AC7. If `propagateFromDocument()` returns an empty `targets` array, `createReviewTasksForPropagation([])` completes without error and creates no tasks. Manual test required: trigger propagation on a document with no downstream dependents, confirm task count is unchanged.

AC8. The `propagation.ts` file contains no logic changes — only the `export` keyword may have been added to the `PropagationTarget` interface if it was not already exported. Manual test required: inspect `artifacts/api-server/src/lib/propagation.ts` and confirm `detectPropagationTargets` and `propagateFromDocument` function bodies are unchanged.

AC9. `GET /api/tasks` returns the auto-created review tasks with `linked_document_name` correctly populated (not null), matching the flagged document's name field.
