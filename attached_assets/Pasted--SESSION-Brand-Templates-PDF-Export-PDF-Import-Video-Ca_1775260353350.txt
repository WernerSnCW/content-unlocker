# BUILD A — Brand Template Engine + PDF Export

## Overview

Generated documents currently exist as plain text in the database.
This build adds a brand layer — HTML/CSS templates per document
type — and a PDF export that applies Unlock's brand to any
promoted document. The output is send-ready: agent downloads the
PDF and attaches to email or sends directly.

---

Populate it with the values extracted from the brand guidelines:

Replace placeholder values with the actual extracted values.
Export BRAND as the single source of truth for all templates.

---

## Step 2 — Create the template engine

Create a new file: api-server/src/lib/templates/index.ts

The template engine takes a document record and returns styled
HTML ready for PDF conversion.

```typescript
export type DocumentTemplate =
  'one-pager'      |   // 100-series — single A4 page
  'three-pager'    |   // 110-series — three A4 pages
  'pack'           |   // 120/130-series — multi-page pack
  'briefing'       |   // bespoke briefing (e.g. Duncan Stewart)
  'explainer'      |   // standalone explainer document
  'default'            // fallback for any unrecognised type

export function getTemplate(
  document: DocumentRecord,
  template?: DocumentTemplate
): string
```

Template selection logic:
- If template is explicitly passed: use it
- If file_code starts with '10': use 'one-pager'
- If file_code starts with '11': use 'three-pager'
- If file_code starts with '12' or '13': use 'pack'
- If file_code starts with '18': use 'briefing'
- Otherwise: use 'default'

Each template returns a complete HTML string with:
- Embedded CSS using BRAND constants
- Unlock logo in the correct position (use text "UNLOCK" in
  the brand font as a placeholder if no logo file is available)
- Document title and metadata header
- Styled body content (convert markdown-style content to HTML)
- Page footer with document reference and date
- Print-ready CSS (@page rules, page-break handling)

The 'one-pager' template must fit on a single A4 page.
All other templates handle multiple pages automatically.

---

## Step 3 — Create PDF export endpoint

Create a new endpoint: POST /api/documents/:id/export-pdf

This endpoint generates a branded PDF from a document record.

Implementation:
- Call getTemplate() with the document record
- Use puppeteer or a similar headless browser library to
  convert the HTML to PDF
- If puppeteer is not available in the Replit environment,
  use an alternative: html-pdf-node, @sparticuz/chromium,
  or wkhtmltopdf if installed
- Return the PDF as a downloadable response:
  Content-Type: application/pdf
  Content-Disposition: attachment; filename="{document_title}_{date}.pdf"

If PDF generation fails, return 500 with a clear error message.
Do not return a partial PDF — fail cleanly.

Request body (optional):
```typescript
{
  template?: DocumentTemplate   // override auto-selected template
}
```

# BUILD B — Finished PDF Import into Registry

## Overview

Existing designed PDFs (produced outside the tool) can be
imported directly into the document registry. The PDF text is
extracted for recommendation matching. The original file is
stored and available for download alongside the generated
branded version.

---

## Step 1 — File storage for documents

Add a file storage layer to the documents table:

Add columns to the documents schema:
```typescript
{
  source_pdf_path: string | null    // path to the stored original PDF
  source_pdf_filename: string | null // original filename
  source_pdf_imported_at: timestamp | null
}
```

Create a directory in the project for stored PDFs:
artifacts/api-server/documents/pdfs/

---

## Step 2 — PDF import endpoint

Create a new endpoint: POST /api/documents/import-pdf

Accepts multipart/form-data with:
- file: the PDF file (required)
- name: document display name (required)
- file_code: document code (optional — auto-assign if not provided)
- tier: 1, 2, or 3 (required)
- persona_relevance: array of archetypes (optional)
- stage_relevance: array of pipeline stages (optional)
- notes: any notes about the source document (optional)

Processing:
1. Extract text from the PDF using pdfjs-dist or pdf-parse
2. Save the PDF file to artifacts/api-server/documents/pdfs/
3. Create a document record with:
   - content: extracted text
   - source_pdf_path: the saved file path
   - source_pdf_filename: original filename
   - lifecycle_status: CURRENT
   - review_state: REQUIRES_REVIEW
     (imported PDFs need QC review before being treated as CLEAN)
   - A DOCUMENT_IMPORTED changelog entry

Response:
```typescript
{
  document_id: string
  name: string
  content_length: number    // characters extracted
  source_pdf_path: string
  review_state: "REQUIRES_REVIEW"
  message: "PDF imported. Run QC before using in recommendations."
}
```

---

## Step 3 — Download original PDF endpoint

Create a new endpoint: GET /api/documents/:id/source-pdf

Returns the original imported PDF as a download.
If no source PDF exists for this document, return 404.

Content-Type: application/pdf
Content-Disposition: attachment; filename="{source_pdf_filename}"

---

# BUILD C — Video Catalogue Stub

## Overview

Creates the video record type in the registry with the correct
schema. No video content yet — records will be added when
scripts are ready. The recommendation engine is made aware of
the video type so it can surface videos alongside documents
when records exist.

---

## Step 1 — Video record schema

Add a new table: videos

```typescript
videos {
  id:                string     // e.g. "vid_001"
  title:             string
  description:       string
  script_content:    string | null   // transcript/script text
  duration_seconds:  integer | null
  format:            string     // e.g. "portrait_mp4", "landscape_mp4"
  send_method:       string     // e.g. "whatsapp", "email_link"
  persona_relevance: jsonb      // array of archetypes
  stage_relevance:   jsonb      // array of pipeline stages
  objections_addressed: jsonb   // array of objection names
  lifecycle_status:  string     // DRAFT | CURRENT | ARCHIVED
  review_state:      string     // CLEAN | REQUIRES_REVIEW | BLOCKED
  created_at:        timestamp
  updated_at:        timestamp
}
```

---

## Step 2 — Video CRUD endpoints

Create basic endpoints:

GET /api/videos — list all CURRENT videos
POST /api/videos — create a new video record
GET /api/videos/:id — get single video with full script
PATCH /api/videos/:id — update video metadata or script
POST /api/videos/:id/promote — set lifecycle_status: CURRENT

---

## Step 3 — Include videos in rank endpoint

Update POST /api/recommendation/rank to include videos in its
results when CURRENT videos exist that match the persona and stage.

Add to the rank response:
```typescript
{
  ranked_documents: [...],    // existing
  recommended_videos: Array<{
    video_id: string
    title: string
    description: string
    duration_seconds: number | null
    send_method: string
    relevance_reason: string
    persona_match: boolean
    stage_match: boolean
  }>
}
```

If no CURRENT videos exist or none match the persona/stage,
return recommended_videos: [] — do not error.

---

## Step 4 — Show videos in recommend flow frontend

Update the rank results step in the recommend flow to show
recommended videos in a separate section below ranked documents.

Each video card shows:
- Title and description
- Duration (if set)
- Send method badge: "📱 Send via WhatsApp"
- Relevance reason
- "View Script" button (expandable — shows script_content)

If recommended_videos is empty, do not show the section.
