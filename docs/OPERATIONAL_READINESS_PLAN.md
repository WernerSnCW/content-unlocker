# Content Unlocker: Operational Readiness Plan (v4)

## Context

The Content Unlocker works. Transcript analysis, belief tracking, persona detection, document recommendation, content generation, campaigns, compliance, and document lifecycle are all functional. But the tool is **manual-first**: transcripts must be uploaded by hand, intelligence must be generated by clicking a button, beliefs must be analyzed separately, and there's no connection to the tools where the actual operation happens (Pipedrive, Aircall, ActiveCampaign).

Two loops must close:

1. **Intelligence loop:** transcript → analysis → intelligence → beliefs → content recommendation. Currently manual; needs to be automatic. And intelligence must **accumulate** across multiple conversations, not overwrite.
2. **Operational loop:** what happened on the call → what happens next → who to call tomorrow. Currently doesn't exist at all.

And one fundamental shift: **the belief map, not the pipeline stage, drives every decision** — what content to send, what to ask, whether someone is ready. Pipeline stages track operational progress (has Pack 1 been sent?). Belief states track investor readiness (do they understand the risk mechanics?). Content routing listens to beliefs. Stages are just the wrapper.

**Scale:** 25K+ leads in Pipedrive, processing a few hundred per day through the pipeline.

### 0.3 — Fix intelligence generation: merge, don't overwrite

**Problem:** The current intelligence generation prompt sends only the transcript + operator notes to Claude and gets a fresh profile back. Each call overwrites the previous profile. Call 1 surfaces hot_button = "family_security". Call 2 discusses something else. Hot button gets reset.

**Fix:** Pass the existing intelligence profile as context in the Claude prompt. Add explicit instructions:

```
EXISTING INTELLIGENCE PROFILE:
${JSON.stringify(existingIntelligence)}

INSTRUCTIONS:
- This is a PROGRESSIVE analysis. The investor has had previous conversations.
- PRESERVE all existing intelligence unless the new transcript explicitly contradicts it.
- MERGE new SPIN signals with existing ones — do not replace.
- If hot_button was already identified, only change it if the new transcript provides stronger evidence for a different one. Increase confidence if the same hot button is confirmed.
- If qualification fields were already set, only change if explicitly contradicted.
- For each field you update, explain what new evidence drove the change.
- Return the COMPLETE merged profile, not just new findings.
```

**File:** `artifacts/api-server/src/lib/intelligenceService.ts`

### 0.4 — Fix belief analysis: preserve established beliefs

**Problem:** The belief analysis prompt receives only the raw transcript and belief list — no existing states. Claude analyses fresh, which means a belief established in call 1 could be marked UNKNOWN in call 2 simply because it wasn't discussed.

**Current state:** The system already has a partial safety net — only HIGH/MEDIUM confidence updates are persisted. But the prompt should be explicit about preservation.

**Fix:** Pass existing belief states as context. Add instructions:

```
EXISTING BELIEF STATES FOR THIS INVESTOR:
${existingBeliefs.map(b => `${b.belief_id}: ${b.state} (evidence: ${b.evidence || 'none'})`).join('\n')}

INSTRUCTIONS:
- This is a PROGRESSIVE analysis across multiple conversations.
- If a belief is already ESTABLISHED, it stays ESTABLISHED unless the new transcript contains explicit evidence of reversal (investor changed their mind, expressed doubt about something previously confirmed).
- If a belief is PARTIAL, look for signals that would upgrade to ESTABLISHED or downgrade to ABSENT.
- If a belief is UNKNOWN, look for any new signal.
- Only return beliefs where the NEW transcript contains relevant signals. Omit beliefs with no new evidence — they retain their current state.
- For each belief you report, cite the specific quote or exchange from THIS transcript.
```

**File:** `artifacts/api-server/src/lib/beliefAnalysisService.ts`

### 0.10 — Conversations table (generic transcript source)

**Problem:** The plan originally had an Aircall-specific `aircall_calls` table. But transcripts come from multiple sources: Aircall (cold calls), Google Meet (demos/follow-ups), manual upload (fallback). The analysis pipeline doesn't care where the transcript came from.

**New file:** `lib/db/src/schema/conversations.ts` (rename from the existing chat conversations table or use a new name like `lead_conversations`)

```
lead_conversations: id (UUID), lead_id (FK),
  source (aircall/google_meet/manual/phone),
  source_id (text, nullable) — external ID (aircall_call_id, meet_recording_id),
  direction (inbound/outbound/meeting),
  participants (JSONB) — [{name, role: agent/investor/adviser}],
  duration_seconds (integer),
  started_at (timestamp), ended_at (timestamp),
  recording_url (text),
  transcript_text (text),
  transcript_status (awaiting/transcribing/ready/failed/not_applicable),
  transcript_summary (text) — Claude-generated summary of this conversation,
  tags (JSONB, default []),
  outcome (text) — interested/no-interest/no-answer/callback-requested/meeting-booked/not-now,
  outcome_processed (boolean, default false),
  pipeline_status (pending/queued/running/completed/failed),
  pipeline_result (JSONB),
  pipeline_error (text),
  beliefs_before (JSONB) — snapshot of belief states before this conversation was analysed,
  beliefs_after (JSONB) — snapshot after analysis (enables "what did this conversation change?"),
  intelligence_delta (JSONB) — what changed in the intelligence profile,
  created_at
```

This replaces the Aircall-specific `aircall_calls` table. Aircall webhooks write here with `source = 'aircall'`. Google Meet transcripts write with `source = 'google_meet'`. Manual uploads write with `source = 'manual'`.

### 0.11 — Conversation summary generation

**Problem:** Progressive intelligence needs summaries of previous conversations as context for the next analysis. Without summaries, Claude would need the full text of all previous transcripts, which won't fit in context for long-running leads.

**Addition to the analysis pipeline:** After the main analysis completes, generate a 3-5 sentence summary of the conversation focused on: what was discussed, what beliefs were surfaced, what objections were raised, what was agreed for next steps. Store in `lead_conversations.transcript_summary`.

**Future conversations** receive previous summaries as context:
```
PREVIOUS CONVERSATION SUMMARIES:
Call 1 (2026-03-15, Aircall): Discussed portfolio structure. Higher-rate taxpayer confirmed.
  Capital available but tied in BTL property. No EIS awareness. Hot button: family security.
Call 2 (2026-03-22, Google Meet): Demo call. Understood the platform.
  U1-U3 established. U4 raised but investor wanted to think about risk.
  Adviser mentioned — accountant at Deloitte, no EIS experience.

CURRENT CALL TRANSCRIPT:
[new transcript here]
```

## Phase 2: Aircall + Google Meet Integration

Receive call outcomes and transcripts from multiple sources, triggering the analysis pipeline.

### 2.1 — Aircall API design constraints

From Aircall's API:
- `call.ended` does NOT include transcript text — only recording MP3 URL (valid 1 hour)
- Transcription requires AI Assist add-on. Without it: download MP3 → Whisper/Deepgram
- Tags arrive AFTER the call ends — must listen for `call.tagged` events
- Key events: `call.ended`, `call.tagged`, `transcription.created`

### 2.2 — Transcription service

**New file:** `artifacts/api-server/src/lib/transcriptionService.ts`

Two paths determined by `AIRCALL_TRANSCRIPTION_MODE` env var:
- **Path A (ai_assist):** Listen for `transcription.created` → fetch transcript → store → trigger pipeline
- **Path B (external):** On `call.ended`, download MP3 immediately → send to Whisper/Deepgram → store → trigger pipeline

### 2.3 — Extract Aircall transcript utilities

**New file:** `artifacts/api-server/src/lib/aircallTranscriptUtils.ts` — extract `isAircallFormat`, `normaliseAircallTranscript`, `identifyAgentLabel` from recommendation route.

### 2.4 — Call outcome processing

**New file:** `artifacts/api-server/src/lib/callOutcomeService.ts`

Maps tags to operational outcomes:

| Tag | Action |
|-----|--------|
| interested | Set stage = "Called", trigger warm nurture recommendation |
| no-interest | Set stage = "Closed - No Interest", stop all outreach, clear Pipedrive label |
| meeting-booked | Set stage = "Demo Booked", stop outreach, generate confirmation content |
| no-answer | Increment call_attempts. If >=3: downgrade to passive. If <3: keep in queue |
| callback-requested | Set callback_date, pause outreach, add to callback queue |
| not-now | Set outreach_paused_until = +28 days. After 28d: re-enter, reset sequence |

Updates lead record, pushes stage to Pipedrive, writes changelog.

**Tag mapping config:** A `TAG_MAPPING` object at the top of callOutcomeService.ts maps Aircall tag strings to internal outcomes. Case-insensitive lookup via `.toLowerCase()` as default, with explicit overrides for non-obvious mappings (e.g., "wants callback" → CALLBACK). Config, not code — prevents debugging on go-live day.

**Re-entry reset:** When `GET /leads/call-queue` finds leads where `outreach_paused_until <= now`, it flips them at query time: clears `outreach_paused_until`, resets `sequence_stage`, sets `last_call_outcome = 're-entered'`, writes changelog. No background job needed — the queue is generated fresh each morning.

### 2.5 — Aircall webhook handler

**New file:** `artifacts/api-server/src/routes/aircall/index.ts`

**`call.ended`:** Store in `lead_conversations` (source = 'aircall'), match by phone, process tags if present, initiate transcription.

**`call.tagged`:** Append tag, if disposition tag + not processed: run `callOutcomeService`.

**`transcription.created`:** Fetch transcript, store, fire `runPostTranscriptPipeline` if lead matched.

### 2.6 — Fireflies.ai integration (primary Google Meet transcript source)

Manual transcript paste creates friction that means transcripts won't consistently get logged, which means the belief map doesn't update for exactly the leads that matter most — the ones in active conversations. Automate from the start.

**How it works:**
- Fireflies joins Google Meet calls automatically when configured (caller does nothing different)
- After the meeting, Fireflies transcribes and sends a webhook to the app
- App matches to lead by participant email (check `leads.email`, fall back to `pipedrive_persons.email`)
- Creates `lead_conversations` record with `source = 'fireflies'`
- Fires `runPostTranscriptPipeline` automatically
- If no lead match: surfaces in Call Inbox for manual linking (same pattern as unmatched Aircall calls)

**New route:** `POST /fireflies/webhook`
1. Validate webhook signature via `FIREFLIES_WEBHOOK_SECRET`
2. Extract: transcript text, participants, duration, recording URL, meeting title
3. Match participant emails to leads
4. Store in `lead_conversations` with `source = 'fireflies'`
5. If matched: trigger pipeline in background

**File:** `artifacts/api-server/src/routes/fireflies/index.ts`

**Keep manual "Log Meeting" as fallback** for cases where Fireflies wasn't on the call, phone calls, or in-person meetings. The "Log Meeting" action stays on lead detail page: paste transcript, select type (Google Meet / phone / in-person), enter duration. Saves with `source = 'manual'`.

### 2.7 — Google Calendar integration

When a meeting is booked or a callback is scheduled, the investor needs a calendar invite — otherwise they have nothing in their diary. This connects directly to the outcome handling flow.

**Three scenarios:**

**Meeting booked (from "meeting-booked" outcome):**
- Outcome dialog includes date/time picker for the meeting
- App creates a Google Calendar event with investor's email
- Includes Google Meet link automatically
- Sends invite to the investor
- Event appears in caller's Google Calendar
- App knows about meetings today — surfaces in call queue alongside callbacks

**Callback requested (from "callback-requested" outcome):**
- Outcome dialog includes date/time picker for callback
- App creates Google Calendar event, sends invite to investor
- Surfaces in call queue on the callback date

**Follow-up scheduling (new capability):**
- "Schedule Follow-up" action on lead detail page and call prep page
- Caller picks date, time, and type (call/meeting)
- App creates calendar event, sends invite
- Lead appears in call queue on that date
- Covers the gap where caller wants to follow up for reasons that aren't standard Aircall tags

**Integration:** Google Calendar API with OAuth (direct OAuth, not Replit connectors SDK). Needs `calendar.events` read/write scope. Calendar events include a link back to the lead's call prep page so the caller can click through to context before the meeting.

**New file:** `lib/integrations/google-calendar/src/client.ts` — OAuth flow, create/update/delete events, list events for date range
**New file:** `artifacts/api-server/src/lib/calendarService.ts` — createMeetingEvent(), createCallbackEvent(), createFollowUpEvent(), getEventsForDate()

**Modified:** Outcome dialog in call prep / call inbox gains a calendar step: tag outcome → if meeting-booked or callback-requested → date/time picker → calendar event created automatically.

### 4.7 — Content routing audit: remove stage-based logic

**Problem to verify:** The recommendation route currently uses `pipeline_stage_relevance` on documents to filter what's eligible. This should be reviewed:
- **Keep:** Stage as a sanity check (don't send Pack 2 to someone in Outreach — but that should really be gated by F0/F1 beliefs anyway)
- **Remove or deprioritise:** Any logic where stage alone determines content eligibility without checking belief state

This is a review task, not a rewrite. The current code already checks belief gates alongside stage. The task is to ensure belief gates are the primary driver and stage doesn't override them.

5. Beliefs accumulate (don't overwrite) → belief map updates → next content recommended
6. Copy content to AC → log send
7. Dashboard: funnel, wave tracking, errors, belief progress
8. Next morning: call queue regenerated with new priorities

## Additional Suggestions: Making This a High-Quality Intelligence Tool

These are my recommendations for features that would elevate the tool from functional to genuinely powerful. They can be built incrementally after the core phases.

### S1 — Discovery completeness score

After each conversation, score how much new intelligence was gathered (0-100%). Based on: how many new belief signals were detected, how many qualification fields were filled, whether hot button was identified or confirmed, whether SPIN fields were populated. Display on the conversation record and aggregate per agent.

**Value:** Lets management see which callers are actually doing discovery vs just going through the motions. A call that moves 3 beliefs and identifies a hot button scores higher than one that just confirms the investor's name.

### S2 — Belief journey analytics

Track how beliefs progress over time across the lead population. Dashboard showing:
- Which beliefs are hardest to establish (lowest ESTABLISHED rate)?
- Average conversations to establish each belief
- Which beliefs are most commonly BLOCKED vs ABSENT?
- Belief state distribution by persona cluster

**Value:** Identifies systematic weaknesses. If U4 (risk understanding) is ABSENT for 80% of leads after the first call, maybe the script needs work or the risk explainer content isn't landing.

### S3 — Content effectiveness tracking

When a document is sent and a subsequent conversation moves a belief from PARTIAL to ESTABLISHED, link the two events. Over time, build a map of: "Sending document X moves belief Y from ABSENT/PARTIAL to ESTABLISHED Z% of the time."

**Value:** Data-driven content strategy. If the EIS risk explainer (doc 150) establishes U4 only 30% of the time, maybe it needs rewriting. If the IHT case studies (doc 160) establish L1 85% of the time, it's working well.

### S4 — Objection memory across conversations

Track objections raised in each conversation with resolution status. An objection raised in call 1 should be visible in call 2's prep. Mark as: raised, addressed, resolved, recurring.

**Value:** Callers don't re-raise resolved objections. Recurring objections (same objection across multiple leads) surface systematic issues that content or product should address.

### S5 — Smart next action engine

Currently, the "next best action" is basic. Upgrade to a multi-step recommendation:
1. **Conversational goal:** "Establish U4 — ask about risk comfort"
2. **Content to send if unresolved:** "Send doc 150 (EIS risk explainer)"
3. **Gate check:** "U4 must be established before any risk framing or Pack 1"
4. **Escalation trigger:** "If U4 stays ABSENT after 2 attempts, flag for senior review"

**Value:** Turns the tool from "here's what to send" into "here's what to do next and why."

### S6 — Pre-call briefing card (auto-generated)

Before each call, Claude generates a 30-second briefing based on the lead's full context. Something like:

> "Sarah is a Growth Seeker (high confidence). She's a higher-rate taxpayer with capital available. Her hot button is freedom — she wants to stop working for her employer. She's had 2 conversations so far. U1 and U2 are established. U3 is partial — she liked the demo but wanted to check with her husband. U4 has not been raised yet. Today's goal: confirm U3 (did she discuss with husband?), then introduce U4 (risk mechanics). If both establish, she's ready for the founding round conversation."

Cache this per lead, regenerate after each new conversation.

**Value:** The caller walks into every conversation with perfect context in 30 seconds.

### S7 — Lead scoring based on beliefs (not engagement)

Traditional lead scoring uses engagement signals (opened email, clicked link). Belief-based scoring uses: how many qualification gates are green, how many universal beliefs are established, how many persona-specific beliefs are established, how close to F0 gate opening.

Formula: `belief_score = (qualification_gates_green / 4) * 0.3 + (universal_established / 4) * 0.3 + (cluster_established / 3) * 0.2 + (founding_readiness) * 0.2`

**Value:** Prioritises leads who are genuinely ready over leads who just opened an email. A lead with U1-U4 all ESTABLISHED and F0 PARTIAL is more valuable than one who opened every email but hasn't confirmed tax status.
